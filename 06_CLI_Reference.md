# ⌨️ Раздел 3: Справочник команд терминала (CLI)

Управление агро-контроллером осуществляется посредством текстовых команд через последовательный COM-порт (скорость 115200). В системе поддерживается автодополнение по кнопке `Tab` и история ввода (стрелки вверх/вниз).

> [!WARNING]
> **Защита от спама (Queue Limit):** Во избежание переполнения памяти терминал защищен очередью `MessageQueueManager`. Если вы отправите скриптом более 20-30 команд за секунду, часть из них будет проигнорирована с предупреждением о переполнении очереди. Для автоматизации используйте скрипты с задержкой (delay 50-100ms) между командами.

## 1. Системные команды

| `uptime` | *нет* | Показывает время непрерывной работы платы с момента включения. | `uptime` |
| `task_top` | *нет* | Выводит список активных системных потоков и объем свободной памяти для каждого из них. | `task_top` |

### 📋 Детальные параметры для `pin_setup` по типам устройств:

В зависимости от типа устройства (`<тип>`), команда `pin_setup` принимает различный набор параметров:

| Тип устройства | Описание и синтаксис параметров |
| :--- | :--- |
| **`DHT11` / `DHT22`**<br/>(Температура) | `pin_setup <имя> <DHT11\|DHT22> <gpio> <топик> <мин_темп> <макс_темп> [интервал_мс]`<br/>• *Пример:* `pin_setup temp_t DHT22 4 greenhouse/temp 10.0 35.0 5000` |
| **`DHT11_HUM` / `DHT22_HUM`**<br/>(Влажность воздуха) | `pin_setup <имя> <DHT11_HUM\|DHT22_HUM> <gpio> <топик> <мин_влаж> <макс_влаж> [интервал_мс]`<br/>• *Пример:* `pin_setup hum_t DHT22_HUM 4 greenhouse/humidity 20.0 80.0 5000` |
| **`RELAY` / `DIGITAL_OUT`**<br/>(Реле и выходы) | `pin_setup <имя> <RELAY\|DIGITAL_OUT> <gpio> <топик> [начальное_состояние (0\|1)]`<br/>• *Пример:* `pin_setup pump RELAY 2 farm/pump 0` (изначально выключено) |
| **`ANALOG_IN`**<br/>(Аналоговые датчики с защитой) | `pin_setup <имя> ANALOG_IN <gpio> <топик> <порог_мин> <порог_макс> [интервал_мс] [powerPin] [powerStabilizeMs]`<br/>• `powerPin` — GPIO питания датчика (255 или `off` — не используется).<br/>• `powerStabilizeMs` — задержка стабилизации в мс (по умолчанию 100).<br/>• *Пример:* `pin_setup soil ANALOG_IN 35 farm/soil 1000 3000 30000 25 200` |
| **`DIGITAL_IN`**<br/>(Кнопки и входы) | `pin_setup <имя> DIGITAL_IN <gpio> <топик> [интервал_мс] [buttonMode (0\|1\|2)] [debounceMs] [maxHoldMs]`<br/>• `buttonMode` — режим работы: `0` (обычный уровень), `1` (переключатель Toggle), `2` (кнопка без фиксации Momentary).<br/>• `debounceMs` — антидребезг в мс (по умолчанию 20 мс для быстрой реакции).<br/>• *Пример:* `pin_setup btn DIGITAL_IN 19 farm/button 20 2 50 5000` |
| **`SERVO`**<br/>(Сервоприводы) | `pin_setup <имя> SERVO <gpio> <топик> [начальный_угол] [мин_угол] [макс_угол] [скорость] [плавность (0\|1)] [автоотключение (0\|1)] [задержка_отключения_мс]`<br/>• *Пример:* `pin_setup win SERVO 5 farm/window 90 0 180 80 1 1 1000` |

### 🧮 Использование математических выражений в правилах:
Для сервоприводов в поле `[payload]` команды `rule_add` можно задавать динамические выражения. Система автоматически масштабирует и передает значение датчика на сервопривод:
*   `{value}` — использовать значение датчика напрямую (например, угол равен температуре).
*   `{value*2}` — умножение. Если температура 25°C, сервопривод откроется на 50°.
*   `{value+30}` — смещение.
*   `{100-value}` — инверсия (обратная связь).
*   `{value<20?30:150}` — тернарное условие (если температура ниже 20, установить угол 30°, иначе 150°).

*Пример команды:* `rule_add vent "Вент" temp > 25 set_servo window_servo {value*3}`

### ⚙️ Продвинутая адаптация сети (`mesh_advanced`):
Команда позволяет адаптировать радиопередатчик к внешним условиям среды:
*   `mesh_advanced status` — вывести отчет о текущей адаптации.
*   `mesh_advanced weather <clear\|rain\|snow\|auto>` — адаптация под погоду. Режим `rain` (дождь) автоматически повышает мощность передатчика для компенсации затухания радиоволн в каплях воды.
*   `mesh_advanced field <on\|off\|auto>` — полевой режим. При `on` оптимизирует дальность передачи на открытых пространствах.
*   `mesh_advanced distance <short\|long\|auto>` — расстояние между платами. Режим `long` увеличивает интервалы ожидания ответов для работы на дистанциях свыше 500 метров.
*   `mesh_advanced threshold <dBm>` — отсекать слабых соседей с сигналом хуже указанного (от -90 до -30 dBm).

## 5. Бортовой самописец (Черный ящик)

Бортовой самописец непрерывно сохраняет важные метрики жизнедеятельности во внутреннюю память.

| Команда | Аргументы | Описание | Пример |
| :--- | :--- | :--- | :--- |
| `blackbox` | `[status]` | Вывести общий статус самописца и обнаруженные при перезагрузке критические ошибки. | `blackbox` |
| `blackbox rtc` | *нет* | Показать технический снимок системы (память, аптайм, код перезапуска) перед последним сбоем. | `blackbox rtc` |
| `blackbox entries`| `[N]` | Вывести N последних текстовых записей логов (WARN/ERROR/PANIC), сохраненных перед сбоем. | `blackbox entries 10` |
| `blackbox clear` | *нет* | Очистить всю историю логов в оперативной памяти RTC. | `blackbox clear` |
| `blackbox stats` | *нет* | Показать статистику износа секторов Flash-памяти LittleFS. | `blackbox stats` |
| `blackbox lifespan`| *нет* | Расчет срока службы чипа Flash-памяти на основе интенсивности записей. | `blackbox lifespan` |
| `blackbox reset` | *нет* | Сбросить историю падений и счетчик критических ошибок в заголовке. | `blackbox reset` |

### ⚙️ Конфигурация записи логов:
Управление триггерами записи событий в Черный Ящик осуществляется группой команд конфигурации:
*   `blackbox_config show` — вывести текущие пороги и перечень активных триггеров.
*   `blackbox_config capture_warnings <on\|off>` — записывать в Черный Ящик все предупреждения системы (уровень `WARN`).
*   `blackbox_config capture_heap <on\|off>` — авто-запись контекста при падении оперативной памяти ниже 10 КБ.
*   `blackbox_config capture_wifi <on\|off>` — авто-запись при падении сигнала WiFi ниже -70 dBm.
*   `blackbox_config capture_loop <on\|off>` — авто-запись при замедлении выполнения главного цикла платы свыше 100 мс.
*   `blackbox_config reset` — сбросить конфигурацию самописца к заводским настройкам.

### 💾 Режим сохранения резервных копий во Flash-память:
По умолчанию данные логов хранятся в энергонезависимой статической памяти RTC (не изнашивает Flash-память, но сбрасывается при полном отключении питания). Для ответственных узлов можно активировать гибридный режим:
*   `blackbox_flash status` — узнать статус гибридного режима и текущий сектор износа.
*   `blackbox_flash on` — включить периодическое резервное копирование логов из RTC во Flash-память LittleFS.
*   `blackbox_flash off` — отключить гибридный режим (логи хранятся только в RTC).
*   `blackbox_flash backup` — принудительно скопировать текущий буфер логов во Flash-память.
*   `blackbox_flash erase` — стереть архив резервных копий с Flash-памяти платы.
*   `blackbox_persistent [on|off|status]` — включить/выключить циклическое сохранение счетчиков в энергонезависимую память NVS.

## 6. Управление логами и диагностикой

Управление выводом системных сообщений и инструменты самодиагностики.

| Команда | Аргументы | Описание | Пример |
| :--- | :--- | :--- | :--- |
| `loglevel` | `<0-3>` | Установить уровень логирования: `0` — только CRITICAL, `1` — WARNING, `2` — INFO, `3` — DEBUG (все сообщения). | `loglevel 2` |
| `loginfo` | *нет* | Вывести текущий уровень логирования и статус цветового вывода. | `loginfo` |
| `colors` | `<on\|off>` | Включить или выключить ANSI-цвета в выводе терминала. Отключите при работе с терминалами без поддержки ANSI (PuTTY, Minicom). | `colors on` |
| `messages` | `[N\|clear]` | Вывести N последних сетевых сообщений из буфера (`MessageBuffer`), либо очистить буфер. | `messages 20` |
| `log_pause` | *нет* | Вывести статус паузы логов и оставшееся время. | `log_pause` |
| `log_pause resume` | *нет* | Немедленно возобновить вывод логов, снять активную паузу. | `log_pause resume` |
| `log_pause timeout <сек>` | `<0.1-60.0>` | Установить таймаут автоматической паузы в секундах. Значение сохраняется в Flash-памяти. | `log_pause timeout 10` |
| `debug` / `wifi_logs` | `<on\|off>` | Включить/выключить низкоуровневые отладочные логи ESP-IDF (Wi-Fi стек, драйверы). Полезно для диагностики сетевых проблем. | `debug on` |
| `notifications` | `[clear\|quiet\|normal]` | Управление системными уведомлениями. `quiet` — включить тихий режим (только CRITICAL). `clear` — очистить очередь уведомлений. | `notifications quiet` |
| `system_diag` | *нет* | Запустить полную самодиагностику системы: память, конфигурации, статус модулей. | `system_diag` |
| `memory_diag` | *нет* | Расширенная диагностика памяти (DRAM, IRAM, Heap фрагментация). | `memory_diag` |
| `self_reflection` | `<status\|enable\|disable\|force\|report>` | Управление системой саморефлексии. `report` — вывести отчет оптимизации. `force` — принудительная проверка. | `self_reflection report` |
| `stability` | *нет* | Статистика стабильности работы узла (количество ресетов, время наибольшего аптайма). | `stability` |
| `crc_check` | `<status>` | Вывести статус защиты CRC целостности пакетов. | `crc_check status` |
| `queue_status` | *нет* | Показать количество элементов в очереди сообщений. | `queue_status` |
| `queue_clear` | *нет* | Очистить очередь входящих сообщений. | `queue_clear` |
| `clear` | *нет* | Очистить экран терминала (ANSI escape-последовательность). | `clear` |

### 📊 Уровни логирования (`loglevel`)

| Уровень | Значение | Что показывает |
| :---: | :--- | :--- |
| `0` | CRITICAL | Только критические ошибки и паники (минимум шума) |
| `1` | WARNING | CRITICAL + предупреждения (рекомендуется для производства) |
| `2` | INFO | WARNING + информационные сообщения о событиях (рекомендуется при настройке) |
| `3` | DEBUG | Все сообщения включая отладочные (для разработки) |

### 💬 Управление веб-сервером (`webserver`)
*Доступно только при сборке с флагом `ENABLE_WEBSERVER`.*

| Команда | Описание | Пример |
| :--- | :--- | :--- |
| `webserver start` | Запустить локальный PWA Web-интерфейс | `webserver start` |
| `webserver stop` | Остановить веб-сервер | `webserver stop` |
| `webserver status` | Вывести статус веб-сервера (запущен / активен) | `webserver status` |
