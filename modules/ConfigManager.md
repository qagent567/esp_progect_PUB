# ⚙️ Модуль ConfigManager (src/ConfigManager.cpp)

## 1. Обзор модуля и его роль в системе
`ConfigManager` — это подсистема абстракции над энергонезависимой памятью устройства (Flash / LittleFS). Она предоставляет унифицированный интерфейс для сохранения и загрузки всех пользовательских и системных настроек в формате JSON.

**Основная роль:**
*   Чтение/запись конфигураций: Wi-Fi, роли узлов, правила автоматизации (`RuleEngine`), задачи планировщика (`Scheduler`) и настройки оборудования (`PinManager`).
*   **Защита Flash-памяти (Wear-Leveling & Cache):** Реализация асинхронного кэширования записи (Delayed Write) для предотвращения физического износа флеш-чипа при частых изменениях состояния.
*   Контроль целостности данных: вычисление CRC32 для каждого сохраняемого JSON-файла для защиты от повреждений (Corruption).

## 2. Зависимости (Include Graph)
*   `LittleFS` — базовая файловая система ядра ESP32.
*   `ArduinoJson` — сериализация/десериализация.
*   `FreeRTOS (semphr.h, task.h)` — фоновая задача и мьютексы для безопасного многопоточного кэширования.
*   `CRC32` — контроль целостности.

## 3. Анализ защиты от износа памяти (ФАЗА 5: Асинхронное кэширование)
До введения оптимизаций, каждое изменение (например, датчик обновил свое состояние) вызывало немедленную запись файла на диск. Это могло "сжечь" Flash за несколько недель. 
Текущая архитектура использует паттерн **"Dirty Cache" (Отложенная запись)**.

### 3.1. Статический кэш
`ConfigManager` использует статические переменные для кэша (`_dirtyCache`, `_cacheMutex`), что означает, что кэш един для всей системы, независимо от того, сколько экземпляров `ConfigManager` создано.

### 3.2. Алгоритм `_saveJsonToFile`
1.  Сериализует JSON в строку.
2.  Берет мьютекс.
3.  Проверяет, есть ли уже этот файл в кэше `_dirtyCache`. Если есть, и контент идентичен — **игнорирует запись** (0 операций с диском).
4.  Если контент новый, помещает его в словарь `_dirtyCache[path] = jsonStr` и ставит флаг `_hasUnsavedChanges = true`. Запись на диск в этот момент **не происходит**.

### 3.3. Фоновая задача `_flashWriterTask`
При запуске `begin()` создается задача FreeRTOS (на Core 1).
Она бесконечно крутится с минимальным потреблением CPU (`vTaskDelay`).
Если она видит флаг `_hasUnsavedChanges`, она ждет таймаут (`AUTO_SAVE_INTERVAL` = 15 минут) или явный вызов `forceSave()`. Только тогда она физически открывает `LittleFS` и делает `print(content)`. Это объединяет сотни мелких изменений в одну операцию записи, увеличивая ресурс памяти на порядки.

## 4. Построчный анализ логики бизнес-конфигураций

### 4.1. Целостность JSON и CRC32
В каждом JSON-файле на верхнем уровне добавляется поле `"crc"`. 
При записи: сначала сериализуется чистый объект, считается `_calculateCRC()`, затем добавляется поле `"crc"`, и объект сериализуется повторно для записи.
При чтении: извлекается `"crc"`, удаляется из объекта, считается хеш от оставшихся данных. Если они не совпадают — файл считается "битым" (возможно, отключили питание прямо во время записи), и загружаются настройки по умолчанию.

### 4.2. Иерархия настроек
*   **Глобальные:** `/wifi_config.json`, `/mesh_config.json`, `/node_role.json`.
*   **Динамические коллекции (forEachPattern):** Для пинов (`/pins/`), правил (`/rules/`) и задач (`/tasks/`) используется подход "один объект = один файл". Метод `forEachRule` сканирует директорию и применяет callback-функцию к каждому файлу. Это избавляет от необходимости загружать гигантский массив всех правил в оперативную память (OOM Protection), позволяя обрабатывать их последовательно (O(1) по памяти).

## 5. Выводы
Модуль `ConfigManager` представляет собой продвинутый ORM (Object-Relational Mapping) для файловой системы встраиваемого устройства. Внедрение асинхронного кэширования записи (Delayed Write Cache) и строгой CRC-проверки превращает обычную `LittleFS` в высоконадежное хранилище, устойчивое к сбоям питания и преждевременному износу кристалла Flash-памяти.
