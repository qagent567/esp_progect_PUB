# Аппаратные Драйверы (Hardware Drivers)

**Путь к директории:** `include/drivers/`, `src/drivers/`

## Описание архитектуры драйверов
Подсистема драйверов реализует паттерн **"Стратегия" (Strategy)** для изоляции логики работы с конкретным аппаратным обеспечением (датчиками, реле, моторами) от ядра системы. В основе всей подсистемы лежит базовый абстрактный интерфейс `IPinDriver`.

Такой подход позволяет ядру платформы (`PinManager`, `RuleEngine`) взаимодействовать с любым железом через единый стандартизированный интерфейс, не задумываясь о том, как аппаратно опрашивается конкретный пин (аналоговое чтение, I2C, OneWire и т.д.).

## Базовый интерфейс: `IPinDriver`
Каждый драйвер железа обязан реализовать этот интерфейс:
- `setup(const PinConfig& config)`: Инициализация пина (например, вызов `pinMode`).
- `update(unsigned long nowMs, SensorData& outData)`: Циклический неблокирующий опрос.
- `getValue()`: Получение последнего закэшированного значения (без аппаратного опроса).
- `setState(float value)`: Управление актуаторами (включение реле, поворот сервопривода).
- `health_check()`: Самодиагностика датчика на зависания.
- `hardware_reset()`: Программная или аппаратная перезагрузка в случае сбоя.

## Реализованные Драйверы

### 1. `AnalogDriver`
Обеспечивает неблокирующую работу с аналоговыми датчиками (датчики влажности почвы, фоторезисторы, потенциометры).
- Использует АЦП ESP32 (`analogRead`) с маппингом сырых значений (0–4095) в пользовательские единицы (% или иные диапазоны) на основе калибровки в `ConfigManager`.
- **Конечный автомат (FSM):** Логика опроса реализована через три состояния: `IDLE → STABILIZING → performRead`. Это гарантирует неблокирующее ожидание после подачи питания на датчик. Состояние `STABILIZING` ждет истечения `powerStabilizeMs` без задержки (`delay()`), передавая управление обратно в `loop()`.
- **Поддержка управления питанием (`powerPin`):** Если задан `powerPin`, драйвер включает питание датчика непосредственно перед измерением и отключает его сразу после. Это продлевает срок службы резистивных датчиков влажности почвы и снижает общее потребление тока.
- **EMA-фильтрация (α = 0.3):** Вместо простого усреднения используется **Экспоненциальное скользящее среднее** (EMA). Это позволяет эффективно подавлять шумы АЦП без накопления буфера и без динамических аллокаций. Старые измерения весят меньше (`α = 0.3`), новые — больше (`1 - α = 0.7`).

### 2. `DigitalInDriver`
Драйвер для работы с цифровыми датчиками (герконы, кнопки, датчики протечки).
- Считывает состояние `HIGH`/`LOW`.
- Имеет встроенную защиту от дребезга контактов (Debounce) без использования блокирующих задержек (`delay()`).

### 3. `DhtDriver`
Драйвер для цифровых датчиков температуры и влажности (DHT11, DHT22).
- Работает по специфичному 1-Wire протоколу.
- Реализует неблокирующее ожидание ответа датчика, чтобы не тормозить цикл FreeRTOS.
- Содержит логику обработки ошибок CRC и таймаутов, типичных для серии DHT.

### 4. `RelayDriver`
Драйвер управления силовыми реле и транзисторами (актуатор).
- Поддерживает конфигурацию прямой и обратной логики (Low-level trigger vs High-level trigger).
- Обеспечивает мгновенную реакцию на вызовы `setState()`.

### 5. `ServoDriver`
Драйвер для управления сервоприводами.
- Использует аппаратные таймеры ESP32 для генерации ШИМ (PWM) сигнала.
- Позволяет плавно или скачкообразно устанавливать угол поворота (обычно 0-180 градусов).

## Надежность и Отказоустойчивость
Каждый драйвер включает собственную реализацию `health_check()`. Если АЦП возвращает `NaN` или датчик DHT перестает отвечать, система может инициировать аппаратный сброс только проблемного пина (или переключить питание датчика), сохраняя стабильную работу остального узла.
