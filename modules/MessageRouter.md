# 🔀 Модуль MessageRouter (src/MessageRouter.cpp)

## 1. Обзор модуля и его роль в системе
`MessageRouter` — это сверхбыстрый коммутатор (демультиплексор) входящих сообщений. Его главная задача — принять строковый JSON пакет из радиоэфира (от `NetworkManager`), десериализовать его, определить тип (`type`) и мгновенно передать его соответствующему зарегистрированному обработчику.

Модуль спроектирован с упором на максимальную скорость маршрутизации и предотвращение блокировок главного цикла.

## 2. Зависимости (Include Graph)
*   `ArduinoJson.h` — парсинг текстовых сообщений.
*   `HashUtils.h` — библиотека вычисления 32-битного хеша (FNV-1a) для строк, что позволяет избежать медленного `strcmp`.
*   `CoreStabilityManager` — отправка heartbeat-ов (watchdog).
*   `Logger` — журналирование маршрутизации.

## 3. Анализ структур данных и классов
Для достижения O(1) и O(log N) скорости поиска обработчика, `MessageRouter` хранит два массива:
1.  **Массив перечислений (Enum Array):** `MessageHandler _typeHandlers[8]`. Для известных базовых типов (например, `MESSAGE_TYPE_COMMAND` = 1, `MESSAGE_TYPE_BROADCAST` = 2) обработчики лежат в статическом массиве. Обращение происходит за время **O(1)** (`_typeHandlers[type]`).
2.  **Массив хешей (Hash Vector):** `std::vector<HashHandler> _stringTypeHandlers`. Для динамически добавляемых типов топиков строковое имя (`"sensor_temp"`) превращается в 32-битный хеш (`uint32_t`). Вектор отсортирован по значению хеша. Обращение происходит за время **O(log N)** с помощью `std::lower_bound` (бинарный поиск).

Такая структура полностью исключает использование динамической памяти при поиске топика.

## 4. Построчный анализ логики и интерфейсов

### 4.1. Регистрация обработчиков (`registerHandler`)
Модуль поддерживает два типа регистрации:
*   По Enum: `registerHandler(MessageType type, std::function...)`. Вставляет лямбду напрямую в массив по индексу.
*   По строке: `registerHandler(const String& type, std::function...)`. При регистрации вычисляет хеш от строки `type`, ищет его место в отсортированном `_stringTypeHandlers` и вставляет (или перезаписывает) обработчик.

### 4.2. Обработка сообщений (`processMessage`)
Ключевой метод, вызываемый при каждом принятом из эфира пакете.
1.  Вызывает `CoreStabilityManager::getInstance().heartbeat(MANAGER_MESSAGE_ROUTER)`, показывая, что роутер жив.
2.  Парсит строку через `deserializeJson(doc, message)` в статический буфер `StaticJsonDocument<512>`. Использование статического буфера гарантирует отсутствие фрагментации кучи.
3.  Определяет `type`. Если тип базовый (из Enum), вызывает обработчик из `_typeHandlers` (O(1)).
4.  Если тип кастомный (пользовательский топик), вычисляет хеш строки `type` и бинарным поиском (`std::lower_bound`) находит нужный `HashHandler` (O(log N)).
5.  Вызов обработчика обернут в блок `try/catch`. Если в обработчике возникнет исключение `std::exception`, роутер перехватит его, запишет ошибку в `Logger` (`_errorCount++`) и не даст ядру ESP32 уйти в Panic.

### 4.3. Обработка неизвестных сообщений (`_handleUnknownMessage`)
Если тип пакета не зарегистрирован ни в одном массиве, вызывается `_handleUnknownMessage`. Пакет сериализуется обратно (для отладки) и выводится в лог уровня WARNING, после чего счетчик ошибок увеличивается.

## 5. Аппаратные и системные интерфейсы
Модуль строго ограничен логической обработкой и не работает напрямую с аппаратной периферией. Его главная "системная" черта — жесткий контроль памяти (стастический JSON-документ) и быстрая математика (бинарный поиск хешей).

## 6. Узкие места (Bottlenecks)
*   Размер буфера `StaticJsonDocument<512>`. Если из Mesh-сети придет пакет больше 512 байт, парсер `deserializeJson` вернет ошибку `NoMemory`. Это защита от OOM, но она накладывает ограничения на объем пересылаемых данных в одном пакете.
