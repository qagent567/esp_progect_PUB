# 🌐 Модуль WebServerManager (src/WebServerManager.cpp)

## 1. Обзор модуля и его роль в системе

> [!IMPORTANT]
> **🆕 Ключевая новинка v4.0.4** — в этой версии добавлен полноценный PWA-веб-интерфейс. Теперь управлять узлом AgriSwarm можно через браузер с любого устройства — телефона, планшета или ноутбука — без USB-кабеля, без Serial-монитора, без интернета.

`WebServerManager` — это асинхронный HTTP-сервер, предоставляющий графический веб-интерфейс (Web UI) и REST API для управления узлом AgriSwarm.
Поскольку ESP32 обладает крайне ограниченным объемом оперативной памяти (RAM) и встроенной флеш-памяти, классические подходы к хостингу веб-сайтов здесь неприменимы. `WebServerManager` использует агрессивные методы оптимизации: предварительное аппаратное сжатие ресурсов (Pre-compressed GZIP) и динамическое управление состоянием (Standby Mode).

**Основная роль:**
*   Хостинг графического веб-интерфейса (PWA - Progressive Web App).
*   Предоставление REST API (`/api/status`, `/api/pins`, `/api/rules`) для связи фронтенда с ядром системы.
*   Поддержка Server-Sent Events (SSE) для трансляции логов и состояний пинов в реальном времени.
*   Поддержка OTA-обновлений прошивки прямо через браузер (`/api/update`).

### 🎨 Возможности веб-интерфейса (UI Features)

| Функция | Описание |
| :--- | :--- |
| 🗺️ **Интерактивная карта пинов ESP32** | 2D-визуализация платы прямо в браузере. Каждый назначенный пин подсвечивается своим цветом (реле — зелёный, аналог — оранжевый, DHT — фиолетовый, серво — розовый). При наведении — тултип с текущим значением |

## 2. Ключевые инновации и архитектура

### 2.1. Аппаратное сжатие сайта и Python-конвейер сборки (PROGMEM GZIP)
В отличие от обычных решений на ESP32, где файлы веб-интерфейса хранятся во внутренней файловой системе LittleFS или компилируются вручную, в AgriSwarm встроен автоматический конвейер сборки ресурсов:
1. **Скрипт сборки:** В проекте используется Python-скрипт `generate_web_resources.py`.
2. **Исходники:** Он считывает исходные файлы фронтенда из папки `src/web`:
   * `index.html` (оригинальный размер: **247.6 КБ**)
   * `manifest.json` (оригинальный размер: **913 байт**)
   * `sw.js` (оригинальный размер: **1.8 КБ**)
3. **Компрессия на лету:** Скрипт сжимает эти файлы алгоритмом GZIP. Сжатый размер `index.html` составляет всего **~70-80 КБ** (экономия flash-памяти более 70%!).
4. **Генерация C++ ресурсов:** Сжатые бинарные данные преобразуются в шестнадцатеричные C-массивы и записываются в заголовочный файл `include/WebRes.h` с макросом `PROGMEM`.
5. **Нулевая нагрузка на CPU при отдаче:** При обращении клиента к веб-серверу ESP32 **не выполняет операцию сжатия или распаковки**. Она просто перенаправляет поток байт из Flash напрямую в TCP-сокет со встроенным HTTP-заголовком `Content-Encoding: gzip`. Распаковкой занимается браузер клиента (мобильного телефона или ПК), что полностью снимает вычислительную нагрузку с контроллера.

### 2.2. Архитектура сбережения ресурсов «10/90» и Режим Ожидания (Standby Mode)
В условиях реальной эксплуатации в полевых хозяйствах и теплицах веб-интерфейс используется максимум **10% времени** — исключительно для первоначальной настройки, калибровки датчиков или просмотра текущих логов. Остальные **90% времени** плата функционирует автономно, обмениваясь mesh-пакетами и выполняя локальные правила.

Держать веб-сервер постоянно в активном состоянии с запущенным вещанием логов нерационально: открытые SSE-соединения и буферизация логов забивают RAM и создают фоновую нагрузку на CPU, что может нарушить критический тайминг mesh-сети.

Поэтому `WebServerManager` реализует двухрежимную архитектуру:

1. **Режим ожидания (Standby Mode - по умолчанию):**
   * При старте системы `SystemOrchestrator` запускает веб-сервер, но он находится в "холодном" состоянии ожидания (`_isActive == false`).
   * В этом режиме к ядру системы **не подключаются** ресурсоемкие слушатели логов (`Logger::setLogListener(nullptr)`) и датчиков (`PinManager::setWebPinCallback(nullptr)`), что экономит 100% процессорного времени.
   * При попытке зайти по IP-адресу платы сервер отдаёт простую страницу-заглушку `standby_html` весом всего 3 КБ, которая хранится во Flash и не требует RAM под буферы данных.

2. **Загрузка «Тяжелого» UI по требованию (Active Mode):**
   * Тяжёлое SPA-приложение (248 КБ) **не выгружается** в сеть при первой загрузке.
   * Пользователь нажимает кнопку **"Активировать интерфейс"**, что инициирует POST-запрос на `/api/server/start` и переводит сервер в режим `Active`.
   * Только теперь выделяется память под потоковые SSE-каналы, подключаются обратные вызовы логов и пинов, а браузер клиента перезагружается и скачивает сжатый GZIP-сайт.

3. **Автоматическое засыпание (Auto-Sleep / Idle Timeout):**
   * Метод `WebServerManager::update()` регулярно проверяет время последнего запроса.
   * Если пользователь закрыл вкладку или не проявляет активности в течение **5 минут** (`SERVER_TIMEOUT_MS = 300000`), сервер автоматически переходит обратно в Standby.
   * SSE-соединения разрываются, слушатели логов/датчиков деактивируются, а занятая под них RAM освобождается.

4. **Полная деаллокация из RAM:**
   * С помощью команды `stopServer()` объект `AsyncWebServer` можно полностью удалить из памяти (`delete _server; _server = nullptr;`), вернув системе всю занятую им оперативную память.

### 2.3. Автономность без сторонних зависимостей (Zero-Dependency)
Благодаря встроенному оптимизированному веб-серверу плата AgriSwarm полностью автономна:
* **Независимость от облака:** Все файлы сайта и API находятся внутри самой платы, интернет не требуется.
* **Независимость от приложений:** Не нужно писать отдельные программы для ПК или мобильных телефонов. Для настройки нужен только любой браузер.
* **Свобода ресурсов:** Решение даёт пользователю полноценный интерфейс, не создавая при этом постоянной нагрузки на МК, что обеспечивает стабильную работу узла месяцами.

### 2.4. PWA (Progressive Web App) и Service Workers
Модуль поставляется с файлами `sw.js` и `manifest.json`. Это превращает веб-интерфейс AgriSwarm в полноценное PWA. 
Пользователь может зайти на сайт устройства с телефона и нажать "Добавить на главный экран". Интерфейс установится как отдельное нативное приложение, кэширующее статику для оффлайн работы (когда телефон подключен к AP-точке ESP32 без доступа в Интернет).

### 2.5. Интерактивный CLI через браузер (`CliSerialRedirect`)
Ядро сервера использует класс `CliSerialRedirect`. Это специальная обертка (поток), которая на лету перехватывает весь вывод стандартного `Serial.print()`. 
Когда пользователь отправляет команду через `/api/cli`, вывод системного `CommandExecutor` перенаправляется не в USB-кабель, а аккумулируется в `String` буфер и возвращается пользователю прямо в браузер. Это дает полноценный терминал управления устройством "по воздуху".

### 2.6. Оптимизация памяти (Zero-Alloc в API)
В версии v4.0.4.2 была проведена ревизия обработчиков API. Вместо использования `DynamicJsonDocument(512)` для генерации ответов (например, в `/api/status`), система переведена на `StaticJsonDocument<512>`. 
* Это переносит создание JSON-объекта из уязвимой кучи (Heap) на быстрый стек (Stack) процессора.
* За счет этого снижается амплитуда скачков фрагментации памяти во время частых опросов веб-интерфейсом состояния узла.

## 3. Взаимодействие с ядром (Rest API)
Модуль предоставляет следующие ключевые эндпоинты:
*   `GET /api/status` — возвращает системную телеметрию (Uptime, RAM, Health, кол-во узлов).
*   `GET /api/pins` — возвращает полный слепок всех датчиков и актуаторов узла.
*   `POST /api/pins/toggle` — позволяет переключить состояние пина. Важно: использует приоритет `PRIO_RULE` или `PRIO_MANUAL`, корректно взаимодействуя с `PinManager` и арбитражем команд.
*   `GET /api/rules` / `POST /api/rules/add` — управление автоматизацией.
*   `POST /api/cli` — проброс консольных команд. Позволяет вводить любые CLI-команды прямо в браузере (использует `CliSerialRedirect` для перехвата вывода `Serial.println` и возврата его в HTTP-ответ).

## 4. Выводы
`WebServerManager` демонстрирует, как можно уместить современное PWA-приложение в микроконтроллер с 320 КБ свободной оперативной памяти. Комбинация GZIP-векторизации и механизма Standby Mode позволяет ядру AgriSwarm оставаться стабильным и не падать от OOM (Out Of Memory) даже при агрессивном использовании веб-интерфейса.
