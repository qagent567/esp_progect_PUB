# 📢 Модуль EventManager (src/EventManager.cpp)

## 1. Обзор модуля и его роль в системе
`EventManager` — это глобальная внутрисистемная шина событий (Event Bus), реализующая классический паттерн проектирования Издатель-Подписчик (Publisher-Subscriber / Pub-Sub). 

**Основная роль:**
*   **Слабая связанность (Loose Coupling):** Исключает жесткие зависимости между модулями. Модулю A (например, драйверу датчика) не нужно знать о существовании модуля B (например, сетевого интерфейса), чтобы передать ему данные. Драйвер просто говорит "Данные изменились", а все заинтересованные модули получают уведомление.
*   Единая точка маршрутизации критических системных событий.

*Примечание:* Не путать с `EventBuffer`. `EventBuffer` хранит историю событий и фильтрует дребезг (аппаратный уровень). `EventManager` — это маршрутизатор вызовов (программный уровень). Впрочем, они работают совместно.

## 2. Анализ структур данных

### 2.1. Полезная нагрузка (`SystemEventPayload`)
Универсальная структура (Variant-подобная), способная переносить различные типы данных:
*   `type`: `SYS_EVENT_SENSOR_DATA`, `SYS_EVENT_PIN_CHANGE`, `SYS_EVENT_NODE_CONNECTED` и др.
*   `sourceHash`: 32-битный FNV-1a хеш источника (оптимизация скорости).
*   `floatValue`, `boolValue`, `stringValue` (указатель) и `void* extraData`.
*   *Ограничение:* `stringValue` имеет время жизни, равное времени исполнения вызова `publish()`. Это сделано намеренно (отказ от аллокации `String` внутри шины), поэтому подписчикам запрещено сохранять этот указатель для асинхронного использования.

### 2.2. Хранилище подписок (`_subscribers`)
Использует стандартный `std::map`:
* **Таблица подписчиков**:
  * Контейнер `std::map`, связывающий тип системного события (`SystemEventType`) со списком подписчиков (`std::vector<EventSubscriber>`).
Ключом является тип события, а значением — вектор `std::function` (лямбда-функций). 

## 3. Анализ логики работы

### 3.1. Подписка (`subscribe`)
Любой менеджер в методе `begin()` может подписаться на событие:
* **Пример подписки**:
  * Метод `subscribe` принимает тип события (например, `SYS_EVENT_SENSOR_DATA`) и регистрирует лямбда-функцию для обработки полезной нагрузки (`SystemEventPayload`).
(Именно так `RuleEngine` привязывается к датчикам, не имея прямой ссылки на драйверы).

### 3.2. Публикация (`publish`)
При вызове `publish()`:
1. Модуль проверяет наличие подписчиков для данного `event.type`.
2. В цикле вызывает каждую зарегистрированную лямбду, передавая ей объект `event` по константной ссылке.

*Сложность операции:* O(1) для поиска вектора по типу (точнее, логарифмическая `map.count`, но типов очень мало), и O(N), где N — количество подписчиков, для вызова функций. Операция полностью синхронна. Блокировок (мьютексов) внутри `publish()` нет.

## 4. Выводы
Модуль представляет собой простую, но невероятно эффективную шину сообщений. Его внедрение позволило разорвать циклические зависимости (Circular Dependencies) в заголовочных файлах C++, сделав архитектуру системы модульной и легко тестируемой. Использование хешей и указателей на строки (`const char*`) сохраняет философию Zero-Alloc в "горячем" пути передачи данных.
