# 🏓 Класс: PingManager

## 1. Назначение и роль в системе
Класс `PingManager` реализует подсистему Keep-Alive (проверка активности) для физических узлов роя. Он отправляет эхо-запросы, отслеживает время отклика (RTT), рассчитывает средние и экстремальные задержки, а также предоставляет функционал непрерывного тестирования линков, аналогичный консольной утилите `ping` в ОС Windows.

## 2. Структуры данных и конфигурация

### 2.1. Статистика пинга (`PingStats`)
Ведет учет параметров конкретного получателя:
*   `minLatency` / `maxLatency` / `avgLatency` — минимальная, максимальная и усредненная задержка (мс).
*   `totalPings` / `successfulPings` / `failedPings` — счетчики запросов.
*   `successRate` — процент успешных доставок эхо-пакетов.
*   `recentLatencies` — кольцевой буфер (вектор) из 10 последних замеров задержки для скользящего анализа.

### 2.2. Запрос пинга (`PingRequest`)
Описывает транзакцию эхо-запроса: `requestId`, `targetNodeId`, `sendTime` отправки и флаг активности `isActive`.

## 3. Режимы и функции работы

### 3.1. Периодический авто-пинг (`enableAutoPing`)
В фоновом режиме каждые 60 секунд (`PING_INTERVAL_MS`) узел отправляет тестовые пакеты всем известным соседям для обновления карты линков и замера задержек.

### 3.2. Непрерывный пинг (Continuous Ping)
Режим, запускаемый оператором через CLI (`ping <nodeId> -t`) для детальной отладки прохождения сигнала при монтаже антенн:
*   **Пресеты частоты:**
    *   `fast` (500 мс) — частый опрос.
    *   `normal` (1000 мс) — ежесекундный опрос.
    *   `slow` (2000 мс) — редкий опрос.
*   Отслеживает количество отправленных и принятых пакетов, мгновенно выводя результаты в Serial-интерфейс или SSE-веб-канал.

### 3.3. Анализ качества связи
Метод `getConnectionQuality()` возвращает дробную оценку качества соединения (от 0.0 до 1.0) на основе уровня RSSI и процента потери пакетов, что используется для авто-маршрутизации.
