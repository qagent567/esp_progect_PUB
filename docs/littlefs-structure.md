# 💾 Архитектура Файловой Системы LittleFS

Флеш-память вашего микроконтроллера ESP32 разделена на логические разделы. Один из наиболее важных разделов — это **LittleFS** (высокоэффективная файловая система, оптимизированная для микроконтроллеров). 

AgriSwarm v4.0.4 использует файловую систему не просто как хранилище «всё в одном», а как модульную базу данных. Каждое устройство, датчик или правило автоматизации сохраняется в виде отдельного компактного JSON-файла. Это предотвращает фрагментацию памяти и зависания системы при частых операциях чтения-записи.

---

## 📂 Дерево папок и файлов LittleFS

Ниже приведена полная структура файлов, которые создаются во встроенной памяти ESP32 в процессе работы:

```text
/ (Корневая папка LittleFS)
├── wifi_config.json            # Настройки Wi-Fi точки доступа и клиента (плюс .bak копия)
├── node_name.json              # Человекочитаемое имя этого узла (плюс .bak копия)
├── node_role.json              # Роль узла: Gateway (шлюз) или Node (клиент) (.bak копия)
├── mesh_config.json            # Параметры авторизации в mesh-сети роя (.bak копия)
├── smart_mesh.json             # Настройки адаптивного подключения и лимитов RSSI
├── mesh_stats.json             # Долгосрочная статистика качества связи соседей
├── algorithms.json             # Список зарегистрированных алгоритмов таймера
├── crash_log.bin               # Бинарный след последнего аварийного завершения (BlackBox)
├── events_0.bin                # Бинарный ротационный буфер логов событий (Wear-Leveling)
├── events_1.bin                # Второй буфер ротации для логов событий
│
├── 📁 config/
│   └── log_pause.json          # Временные настройки паузы вывода логов в CLI
│
├── 📁 pins/                    # Конфигурация подключенных датчиков и реле
│   ├── dht_sensor.json         # Пример файла для конкретного DHT22
│   ├── soil_moisture.json      # Пример файла для аналогового датчика влажности
│   └── relay_pump.json         # Пример файла для модуля реле
│
└── 📁 rules/                   # Правила локальной автоматизации (RuleEngine)
    ├── 12948203.json           # Файл правила (название совпадает с FNV-1a хэшем ID)
    └── 34829012.json           # Файл правила для связи датчика с исполнительным реле
```

## 📑 Описание и примеры конфигурационных файлов

Для понимания того, как система хранит данные под капотом, ниже приведены примеры содержимого ключевых файлов конфигурации.

### 1. ⚙️ Глобальная сеть (`/mesh_config.json`)
Файл содержит реквизиты доступа к беспроводной ячеистой сети (Mesh).

```json
{
  "prefix": "AgriSwarmMesh",
  "password": "SwarmSecretPassword",
  "port": 5555,
  "channel": 6,
  "swarmEnabled": true
}
```

> [!NOTE]
> Флаг `"swarmEnabled": true` переводит устройство в активный режим роя. Если он равен `false`, устройство работает как автономный контроллер без радиоактивности.

---

### 2. 🔌 Конфигурация пинов (`/pins/<имя_пина>.json`)
Каждый пин сохраняется в свой собственный файл. Это гарантирует, что добавление или удаление датчика не потребует полной перезаписи большого общего файла конфигурации оборудования.

#### Пример для DHT22 (`/pins/dht_sensor.json`):
```json
{
  "name": "dht_sensor",
  "type": "DHT22",
  "gpio": 4,
  "params": {
    "readIntervalMs": 10000,
    "offsetTemp": 0.0,
    "offsetHum": 0.0
  }
}
```

#### Пример для аналогового датчика влажности почвы (`/pins/soil_analog.json`):
```json
{
  "name": "soil_analog",
  "type": "ANALOG_IN",
  "gpio": 34,
  "params": {
    "readIntervalMs": 30000,
    "powerPin": 25,
    "stabilizationDelayMs": 150
  }
}
```

> [!TIP]
> Параметры `powerPin` и `stabilizationDelayMs` управляют транзисторным ключом питания датчика влажности. Плата подает ток на пин 25, ждет 150 миллисекунд пока стабилизируются показания на АЦП пине 34, считывает значение и мгновенно выключает питание. Это экономит заряд батареи и защищает контакты от коррозии.

---

### 3. 🤖 Локальные правила автоматизации (`/rules/<id_hash>.json`)
Правила `RuleEngine` также сохраняются раздельно, чтобы обеспечить O(1) скорость доступа по хэшу имени правила (FNV-1a).

#### Пример правила управления поливом (`/rules/27849102.json`):
```json
{
  "id": "irrigation_rule",
  "name": "Автополив при сухости",
  "description": "Полив при низкой влажности",
  "enabled": true,
  "lastTriggered": 0,
  "cooldownPeriod": 5000,
  "triggerCount": 0,
  "condition": {
    "type": 1,
    "source": "soil_analog",
    "value": 30.0,
    "value2": 0.0,
    "enabled": true
  },
  "action": {
    "type": 0,
    "target": "relay_pump",
    "payload": "1.0",
    "enabled": true
  }
}
```
*Если датчик `soil_analog` покажет значение меньше `30.0` (тип условия `1` — `CONDITION_LESS_THAN`), RuleEngine мгновенно выполнит действие типа `0` (`ACTION_SET_PIN_STATE`) для целевого исполнительного устройства `relay_pump` со значением `"1.0"` (Включить).*

---

### 4. 🚑 Стабильность и резервные копии (`.bak` файлы)
При каждой успешной загрузке системы или при вводе команды `pin_save_all` / `rule_save`, AgriSwarm делает зеркальную копию критических файлов:
`wifi_config.bak`, `node_role.bak`, `mesh_config.bak`.

Если во время сохранения настроек пропадет питание (например, разрядится аккумулятор или произойдет бросок напряжения), файл конфигурации может оказаться битым (повреждение структуры JSON).
*   **Как работает автовосстановление:** При следующем старте модуль `CoreStabilityManager` проверяет целостность JSON файлов. Если основной файл поврежден, он автоматически заменяется его резервной копией `.bak`, предотвращая зависание системы.

## 🛠️ Команды CLI для работы с LittleFS

Вам не нужно вручную редактировать эти файлы — это может привести к сбоям. Используйте встроенные команды в консоли (CLI):

*   `fs_info` — Выводит статус диска (общий размер, занято, свободно).
*   `health_check` — Сканирует все конфигурационные файлы на предмет ошибок и повреждений.
*   `blackbox stats` — Показывает износ ячеек флеш-памяти и количество циклов стирания секторов LittleFS.
