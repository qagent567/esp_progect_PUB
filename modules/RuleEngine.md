# &#128640; Модуль RuleEngine (src/RuleEngine.cpp)

## 1. Обзор модуля и его роль в системе
`RuleEngine` — это «Двигатель правил», центральный интерпретатор бизнес-логики проекта AgriSwarm. Если `AlgorithmScheduler` предоставляет высокоуровневые «алгоритмы» и планировщик действий с задержками, то `RuleEngine` — это более жесткая, базовая система реакций (Рефлексы).

**Основная роль:**
*   Мгновенная реакция на изменения данных от локальных или удаленных (через Mesh) датчиков.
*   Оценка простых логических условий (`>`, `<`, `==`, `!=`, `BETWEEN`, `CHANGED`).
*   Связывание событий (срабатывание датчика) с физическими действиями (переключение реле, установка угла сервопривода) без написания C++ кода.

## 2. Анализ структур данных и паттернов (O(1) Hashing)

Класс использует хеширование для ускорения сопоставления событий:
*   Вместо медленного хранения и сравнения строк (`String`), используется `uint32_t idHash`, `sourceHash`, `targetHash`.
*   **Unified SourceState:** Все датчики агрегируются в структуру `SourceState`, хранящуюся в `std::vector<SourceState> _sourceStates`.

    ```cpp
    struct SourceState {
        uint32_t hash;
        float lastValue;
        unsigned long lastChangeTime;
        unsigned long changeFrequency;
        std::vector<uint32_t> ruleHashes; // Список хэшей правил, зависящих от датчика
    };
    ```

*   Поиск `SourceState` по хешу датчика реализован линейным перебором (`_findSourceState` — **O(N)**). Это допустимо: количество уникальных источников данных невелико (обычно < 20 датчиков на узле).
*   Поиск правила по хешу внутри обработчика событий — также линейный (`std::find_if` по `_rules`). Оптимизация через предварительное кэширование `ruleHashes` в `SourceState` обеспечивает быстрый «предфильтр» (не нужно проверять все правила, только подписанные на этот датчик).

## 3. Анализ логики и кэширования

### 3.1. Асинхронная обработка (Event Queue)
Движок имеет два режима: синхронный и асинхронный (`enableAsyncProcessing`).
*   Для **цифровых** пинов (`SENSOR_DATA_DIGITAL`) реакция происходит **синхронно и мгновенно** — вызов идет напрямую в `_processSensorDataSync`.
*   Для **аналоговых** (которые часто меняются) события складываются в `std::queue<AsyncEvent> _eventQueue`.
*   В методе `processAsyncEvents()` события достаются «пачками» (`BATCH_SIZE = 10`), с вызовом `yield()` каждые 5 обработанных событий. Это предотвращает блокировку процессора.
*   Максимальный размер очереди — `MAX_QUEUE_SIZE = 50`. Переполнение логируется как WARN, событие отбрасывается. Использование `std::queue` может приводить к фрагментации кучи при длительной нагрузке (известное ограничение v4.x).

### 3.2. Адаптивное кэширование условий (Adaptive TTL)
Проверка условий вынесена в `_conditionCache` (`std::map<uint32_t, CachedConditionResult>`).
*   Ключ кэша: `uint32_t cacheKey = ruleHash ^ (sourceHash << 1);`
*   Алгоритм: `uint32_t cacheKey = ruleHash ^ (sourceHash << 1);`
*   **Adaptive TTL:** если датчик меняется чаще 10 раз в секунду (`changeFrequency > 10`), TTL кэша = `FAST_TTL_MS = 50 мс`. Иначе = `SLOW_TTL_MS = 1000 мс`. По умолчанию `DEFAULT_TTL_MS = 500 мс`.
*   Цифровые пины (`SENSOR_DATA_DIGITAL`) **не кэшируются** — обрабатываются напрямую каждый раз.
*   Кэш автоматически очищается при превышении 100 записей методом `_cleanupExpiredCache()`.

### 3.3. Исполнение действий (Actions)
Полный перечень поддерживаемых типов действий (`RuleActionType`):

| Тип | Константа | Описание |
| :--- | :--- | :--- |
| &#128268; Пин | `ACTION_SET_PIN_STATE` | Дергает локальный пин через `PinManager` с приоритетом `PRIO_RULE`. |
| &#128246; Сеть | `ACTION_SEND_MESSAGE` | Отправляет команду в Mesh-сеть (через `PriorityMessageQueue`) с `requireAck=true`. |
| &#128221; Лог | `ACTION_LOG_MESSAGE` | Запись в лог. Обрабатывается внешним коллбэком `_actionCallback`. |
| &#9654; Задача | `ACTION_TRIGGER_TASK` | Запускает произвольную CLI-команду через `CommandExecutor`. |
| &#127918; Серво | `ACTION_SET_SERVO` | Вычисляет целевой угол сервопривода через `_evaluateServoExpression`, вызывает `PinManager::setServoAngle` с `PRIO_RULE`. |

### 3.4. Fail-Safe Heartbeat (Восстановление состояния)
Метод `update()` раз в **60 секунд** пробегает по всем включенным правилам, у которых `lastState = true` (условие было выполнено), и **повторно** выполняет их действие. Это гарантирует, что если актуатор (реле, клапан) был выключен по любой причине (сброс питания, FailSafe), правило восстановит правильное состояние без вмешательства оператора.

## 4. Потокобезопасность
Все критические операции с векторами и очередями обернуты в `std::lock_guard<std::recursive_mutex>`. Используется **C++17 стандартный рекурсивный мьютекс** (`mutable std::recursive_mutex _mutex`), а не FreeRTOS-примитив. На ESP32 он реализован поверх FreeRTOS, но отличается от `xSemaphoreTakeRecursive`: **не может вызываться из ISR-контекста** (аппаратных прерываний). Вызов из главного цикла (`loop`) и сетевых обработчиков — безопасен.

## 5. Сохранение правил
*   Каждое правило хранится в **отдельном файле** на LittleFS: `/rules/<idHash>.json`. Это позволяет добавлять и удалять отдельные правила без перезаписи всего массива.
*   При загрузке поддерживается **миграция** со старого формата `/rules.json` (единый файл).
*   Автосохранение реализовано через флаг `_rulesDirty` + Debounce-таймер `SAVE_DEBOUNCE_MS = 30000 мс` (30 секунд). Это исключает «флап» записи при частых обновлениях правил.

## 6. Выводы
`RuleEngine` — это высокооптимизированный конечный автомат. Разделение на Fast Path (цифровые пины, синхронно) и Async Queue (аналоговые датчики), адаптивное кэширование результатов вычислений и Fail-Safe Heartbeat позволяют ему обрабатывать тысячи событий в секунду на 240 МГц ядре ESP32, не вызывая срабатывания сторожевого таймера.
