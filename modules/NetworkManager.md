# 🌐 Модуль NetworkManager (NetworkManager.h)

## 1. Обзор модуля и его роль в системе
Файл `include/NetworkManager.h` описывает интерфейс класса `AgriNetworkManager`. Это центральный компонент Сетевого Стека (Layer 3), отвечающий за инициализацию сетевых интерфейсов, поддержание Wi-Fi соединения и организацию Mesh-сети.

**Основная роль:**
*   Оркестрация работы сетевых интерфейсов ESP32 (STA/AP).
*   Управление жизненным циклом Mesh-соединения (инициализация, heartbeat, обработка отключений).
*   Адаптивное автоподключение и автоматический переход в Host-режим при изоляции узла.
*   Отслеживание валидности ID узла и маршрутизация сообщений на базовом уровне.

*Примечание:* В проекте присутствуют две версии заголовочного файла (NetworkManager.h и AgriNetworkManager.h), которые объявляют один и тот же класс `AgriNetworkManager`. Исторически `NetworkManager.h` мог использовать кастомную библиотеку `AgriSwarmMesh`, а `AgriNetworkManager.h` — `painlessMesh`. Текущий анализ базируется на структуре интерфейса.

## 2. Зависимости (Include Graph)
Модуль является высокоуровневым фасадом и объединяет множество подсистем:
*   `WiFi.h` — базовый стек Wi-Fi ESP32.
*   `AgriSwarmMesh.h` / `painlessMesh.h` — библиотека для организации Wi-Fi Mesh-сети.
*   `ArduinoJson.h` — сериализация/десериализация пакетов.
*   `ConfigManager` — загрузка сетевых настроек (SSID, пароли).
*   `MessageRouter` — маршрутизация входящих/исходящих пакетов.
*   `TrustedNodeManager` — фильтрация неавторизованных узлов.
*   `NodeManager` — поддержание информации о состоянии и роли текущего и соседних узлов.
*   `HostManager` — управление режимом точки доступа (Fallback AP).

## 3. Анализ структур данных и классов

Класс `AgriNetworkManager` реализован как сервис (обычно управляемый `SystemOrchestrator`), хранящий состояние сети и ссылки на другие менеджеры:

*   **Ссылки на другие менеджеры:** `_configManager`, `_nodeManager`, `_messageRouter`, `_trustedNodeManager`, `_hostManager`.
*   **Сетевые объекты:** Экземпляр `_mesh` (AgriSwarmMesh или painlessMesh).
*   **Флаги состояний:** `_isAPModeActive`, `_autoConnectionActive`, `_isConnecting`, `_scanInProgress`, `_isHostMode`.
*   **Таймеры и счетчики:** Отслеживание таймаутов автоподключения (`_autoConnectionStartTime`), heartbeat-ов (`_consecutiveFailedHeartbeats`), таймеры сканирования (`_wifiScanTicker`).
*   **Очереди коллбэков:**
    *   `_pendingConnectionCallbacks` — структура `PendingConnectionCallback` для отложенного выполнения действий после подключения узла (помогает избежать перегрузки процессора).
    *   `_pendingSyncNodes` — отложенная синхронизация для стабилизации соединения.

## 4. Построчный анализ логики и интерфейсов

### 4.1. Инициализация и Жизненный цикл
*   `begin()`: Запускает Wi-Fi, инициализирует Mesh, регистрирует обработчики сообщений (`node_info`, `heartbeat`, `ping`, `pong`) и коллбэки Mesh-сети (`onReceive`, `onNewConnection`, `onDroppedConnection`).
*   `handleClient()`: Ключевой метод, вызываемый в `loop()`. Выполняет:
    1.  Отправку heartbeat для `CoreStabilityManager` (Watchdog).
    2.  Вызов `_mesh.update()`. Имеет форсированный режим (частый вызов) в первые 60 секунд работы для ускорения поиска узлов.
    3.  Управление адаптивным Heartbeat-интервалом (меняется в зависимости от числа узлов в сети).
    4.  Асинхронную обработку `_processPendingConnectionCallbacks()`.

### 4.2. Механизм отказоустойчивости (Self-Healing & Auto-Recovery)
*   **Адаптивный Heartbeat:** Если в сети много узлов, интервал `_sendHeartbeat()` увеличивается, чтобы не забивать эфир. Если узлов мало — уменьшается для быстрого обнаружения соседей.
*   **Отложенные коллбэки (Adaptive Delay):** При массовом подключении узлов (`_onNewConnection`), выполнение тяжелых задач откладывается (метод `_getAdaptiveConnectionDelay()`), чтобы не вызывать срабатывание Task Watchdog (WDT).
*   **Host-режим (Fallback):** Если узел не может найти Mesh-сеть в течение заданного времени (`_autoConnectionTimeout`), он автоматически запускает `_startHostMode()`, создавая свою точку доступа, к которой может подключиться инженер для настройки.

### 4.3. Аппаратные и Системные взаимодействия (Big O)
*   Вызов `_mesh.update()` критичен к задержкам. Время его выполнения зависит от объема принятых/отправленных пакетов, но архитектура спроектирована так, чтобы минимизировать блокировки (O(N), где N — количество пакетов в буфере Wi-Fi/TCP стека painlessMesh).
*   Использование `yield()` внутри `_processPendingConnectionCallbacks()` предотвращает панику ядра из-за занятости Core 1.
*   Проверка валидности `Node ID` (`getValidNodeId`) кэшируется (`NODE_ID_CHECK_INTERVAL = 5000` мс), что дает O(1) амортизированное время доступа вместо постоянного дерганья API `_mesh.getNodeId()`.

## 5. Выводы
Интерфейс `NetworkManager.h` описывает высоконадежный, отказоустойчивый сетевой слой, который способен самостоятельно принимать решения об изоляции, переходе в режим точки доступа и регулировании нагрузки на радиоэфир (адаптивные тайминги и heartbeat). Очереди отложенных коллбэков защищают систему от каскадных сбоев при одновременном переподключении десятков устройств.
