# ⚡ Модуль PriorityMessageQueue (src/PriorityMessageQueue.cpp)

## 1. Обзор модуля и его роль в системе
`PriorityMessageQueue` — это высокооптимизированная очередь исходящих сообщений с приоритетами (Max-Heap) и статическим пулом памяти (Memory Pool). В отличие от простого `MessageQueueManager`, этот модуль является результатом архитектурной оптимизации (ФАЗА 4), направленной на полное исключение фрагментации кучи (Zero-Alloc) в "горячем" контуре отправки.

**Основная роль:**
*   Обеспечить гарантированную отправку критических сообщений (аварии, системные команды) вне очереди обычных логов или телеметрии.
*   Управлять повторными попытками (Retries) отправки.
*   Управлять памятью сообщений (Static Pool) без вызовов `new` / `delete`.

```mermaid
flowchart TD
    Req[Запрос памяти для JSON: acquireDoc] --> CheckStatic{Есть свободный слот в Static Pool?}
    CheckStatic -->|Да| AllocStatic[Пометить слот как занятый]
    AllocStatic --> ReturnDoc[Вернуть doc из Static Pool (0 аллокаций в куче)]
    
    CheckStatic -->|Нет| CheckLimit{Лимит Dynamic Memory < 8 KB?}
    CheckLimit -->|Да| AllocDynamic[new DynamicJsonDocument в heap]
    AllocDynamic --> AddVector[Добавить в список динамических документов]
    AddVector --> ReturnDoc
    
    CheckLimit -->|Нет| OOM[Ошибка: Out Of Memory, отбросить сообщение]
```

## 2. Зависимости (Include Graph)
*   `ArduinoJson.h` — для хранения полезной нагрузки (payload) сообщений.
*   `HashUtils.h` — утилиты для хеширования (хотя напрямую здесь не вычисляются).
*   `Logger` — журналирование состояния очереди и предупреждения о вытеснении сообщений (Eviction).

## 3. Анализ структур данных и пула памяти (Memory Management)

Этот класс — шедевр управления памятью для ESP32:
*   **Статический пул JSON (Static Pool):** `static StaticJsonDocument<PMQ_POOL_DOC_SIZE> s_pool[PMQ_POOL_SIZE];`
    Заранее выделено 10 (`PMQ_POOL_SIZE`) буферов по 512 байт. Это 5 КБ памяти, которая выделяется ОДИН РАЗ на этапе компиляции (BSS segment).
*   **Статический пул сообщений:** `static PriorityMessage s_msgPool[PMQ_POOL_SIZE];`
    Сама структура сообщения тоже выделена статически. Никаких строк `String` внутри! `String type` заменено на `char type[24]` для исключения аллокаций.
*   **Динамический Fallback (Dynamic Pool):** Если статических слотов не хватает, включается `std::vector<DynamicJsonDocument*> _dynamicDocs`. Но он жестко ограничен лимитом `PMQ_MAX_DYNAMIC_MEMORY` (8 КБ). Если лимит исчерпан, модуль вернет ошибку OOM, защищая устройство от падения.

## 4. Построчный анализ логики и интерфейсов

### 4.1. Получение и освобождение памяти (`acquireDoc` / `releaseDoc`)
*   `acquireDoc`: Сначала ищет свободный блок в `s_poolUsed`. Если нашел, помечает занятым и возвращает ссылку (0 аллокаций). Если статический пул занят, проверяет лимит 8КБ. Если лимит не превышен, делает `new DynamicJsonDocument`.
*   `releaseDoc`: Возвращает блок в пул, просто сбрасывая флаг `s_poolUsed[i] = false` и делая `clear()`.

### 4.2. Добавление в очередь (`enqueue`)
Метод имеет сложность O(N log N) из-за сортировки:
1.  Запрашивает свободную структуру `PriorityMessage` из пула (`_acquireMessage()`).
2.  Если пул переполнен, алгоритм ищет сообщения с низким приоритетом (`MSG_PRIORITY_LOW`). Если новое сообщение имеет `MSG_PRIORITY_CRITICAL`, то старое (низкоприоритетное) **вытесняется (Eviction)**, освобождая место. Это гарантирует, что система не потеряет критический алерт из-за спама логами.
3.  Копирует payload в выделенный блок JSON (`acquireDoc`).
4.  Вставляет сообщение в вектор `_queue` и сортирует его функцией `_sortQueue()`.

### 4.3. Алгоритм сортировки (`PriorityMessageCompare`)
Каждый раз при добавлении очередь сортируется:
* **Сравнение приоритетов**:
  * Приоритеты сравниваются по уровню важности (`priority`). Если приоритеты равны, более старое сообщение (с меньшим значением `timestamp`) перемещается вперед.
*Правило:* В начало очереди идут сообщения с высоким приоритетом. Если приоритет равен, то более старое сообщение (меньший timestamp) идет первым (FIFO внутри одного приоритета).

### 4.4. Обработка таймаутов (`update`)
Раз в минуту (`60000` мс) метод `update()` пробегает по всей очереди и удаляет сообщения, которые "застряли" в ней дольше 5 минут (300 000 мс) (например, адресат выключен). Это предотвращает вечную блокировку пула. Сложность: O(N).

## 5. Выводы
`PriorityMessageQueue` представляет собой эталонный паттерн для встраиваемых систем реального времени: **Memory Pooling + Priority Preemption (Вытеснение по приоритету)**. Благодаря отказу от динамической памяти в пользу BSS, этот класс может работать месяцами без единого байта утечки, гарантируя при этом 100% доставку SOS-сигналов и управляющих команд даже при перегруженной сети.
