# ⚡ Модуль ActuatorPublisher (src/ActuatorPublisher.cpp)

## 1. Обзор модуля и его роль в системе
`ActuatorPublisher` — это зеркальный аналог `SensorPublisher`, но предназначенный для **исполнительных механизмов (Актуаторов)**: реле, сервоприводов, помп, клапанов.

**Основная роль:**
*   Рассылка информации о доступных локальных реле (Discovery).
*   Прием и маршрутизация команд управления от удаленных узлов к локальному `PinManager`.
*   Управление отправкой управляющих команд с ГАРАНТИЕЙ доставки (ACK, Retries).

## 2. Архитектура и Структуры данных

Архитектурно класс очень похож на `SensorPublisher`, используя ту же парадигму `Zero-Alloc` с хешами и фиксированными массивами `char[]`:
* **Структура ActuatorInfo**:
  * `nodeId` — 32-битный ID узла в сети.
  * `idHash` — 32-битный FNV-1a хэш для быстрого поиска.
  * `actuatorId` — текстовое имя актуатора (буфер `char[32]`).
  * `isAvailable` — статус активности актуатора.
  * `requiredLevel` — уровень доступа (для защиты критических команд).
Отличие заключается в наличии поля `requiredLevel`. Это закладывает основу для поддержки ACL (Access Control List), предотвращая отправку команд на включение критического оборудования (например, нагревателя) от неавторизованных источников.

## 3. Анализ логики (Командный пайплайн)

### 3.1. Получение команды из сети (`handleActuatorInfo`)
1. Модуль принимает JSON (например, `{"type":"actuator_command", "actuatorId":"pump_1", "command":"ON"}`).
2. Записывает этот актуатор в `_availableActuators` (если его там не было).
3. Обновляет статус в локальных подписках.
*(Примечание: Фактическое выполнение локальной команды происходит в `PinManager::handleRemoteActuatorCommand`, `ActuatorPublisher` лишь обновляет сетевые статусы).*

### 3.2. Отправка команды с гарантией (`sendCommandWithAck`)
Это самая критическая функция модуля (используется при срабатывании правила в `RuleEngine`).
1. Формирует JSON с уникальным `commandId` (комбинация MAC-адреса и `millis()`).
2. Добавляет криптографическую защиту: `MessageCRC::getInstance().addCRC(_jsonBuffer)`. Это защита от повреждения пакета помехами.
3. Регистрирует сообщение в `MeshDeliveryTracker`.
4. Передает в `PriorityMessageQueue` с приоритетом, чтобы команда обогнала обычные пакеты телеметрии в буфере радиомодуля.

### 3.3. Retry-цикл (`update()`)
В главном цикле `update()` модуль управляет массивом `_pendingCommands`:
*   Если ACK не пришел за 5 секунд, команда шлется повторно.
*   **Экспоненциальная задержка:** `delayMs = 300UL << std::min(it->retryCount, 5)`. Каждая следующая попытка ждет в 2 раза дольше (300мс, 600мс, 1200мс...). Это спасает сеть от шторма пакетов, если принимающий узел действительно "умер".
*   **Broadcast Fallback:** Если прямая отправка (`sendMessage`) провалилась (маршрут сломан), модуль делает последнюю отчаянную попытку через `broadcast()` (широковещание на все узлы), надеясь, что пакет найдет путь к цели.

## 4. Выводы
`ActuatorPublisher` реализует надежный (Reliable) транспортный слой для управляющих воздействий. Благодаря интеграции с `MeshDeliveryTracker`, криптографическим CRC и логикой экспоненциального повтора (с переключением на broadcast), система гарантирует доставку команды включения насоса даже при деградации качества связи.
