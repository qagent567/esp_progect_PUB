## 🚀 Быстрый старт (Мини-демо за 2 минуты)

Краткий ориентир для запуска и понимания идеи AgriSwarm.

### 🧠 Ментальная модель
AgriSwarm — это сеть узлов ESP32. Каждый узел:
- читает сенсоры и публикует данные как топики;
- управляет актуаторами и публикует их возможности;
- подписывается на топики других узлов;
- выполняет правила и алгоритмы локально;
- общается через Mesh и работает автономно.

```
[Sensor] -> [Node A] ==mesh== [Node B] <- control -> [Actuator]
              ^  publish         subscribe ^
              |----------- Rule/Algo --------|
```

### 💾 Загрузка прошивки (Binaries)

> [!TIP]
> **Все файлы прошивки (.bin) и программатор находятся здесь:**  
> 👉 **[📂 Открыть папку «---для загрузки прошивки---»](---для загрузки прошивки---)**

### ⚡ Базовые действия
```bash
# Запуск графического прошивальщика (для загрузки скомпилированной прошивки)
# Windows: запустите готовый файл ---для загрузки прошивки---/flash_firmware.exe
# Кроссплатформенный вариант (Python):
python "---для загрузки прошивки---/flash_firmware.py"

# Подключение к консоли (через любой Serial Monitor на 115200 бод)
# После подключения доступны полезные команды:
help
pin_wizard                # Настройка сенсора/актуатора
pin_monitor <sensor>
rule_wizard               # Правило: IF sensor THEN action

# Если есть 2 узла — покажите сеть
node_list
topics_tree
```

### 🗺️ Топологии внедрения
- **Одна теплица:** 1–3 узла (климат + полив)
- **Несколько грядок:** 3–8 узлов (зоны полива, освещение)
- **Поле/сад:** 10+ узлов (сегменты, шлюз, релейные актуаторы)

## 🌟 Ключевые технологические столпы

В отличие от большинства типовых DIY-проектов на Arduino, **AgriSwarm** спроектирован с учетом требований к стабильности промышленных встраиваемых систем (Industrial IoT).

```
 ┌─────────────────────────────────────────────────────────────────────────┐
 │                            AGRISWARM CORE                               │
 └────────────────────┬────────────────────────────────┬───────────────────┘
                      │                                │
        ┌─────────────▼─────────────┐    ┌─────────────▼─────────────┐
        │ Zero-Allocation Hotpaths  │    │  Self-Healing Mesh Рой    │
        │ • Исключены утечки кучи   │    │  • До 20 узлов ESP-NOW    │
        │ • O(1) хэш-маршрутизация  │    │  • Работа без Интернета   │
        └───────────────────────────┘    └───────────────────────────┘
                      │                                │
        ┌─────────────▼─────────────┐    ┌─────────────▼─────────────┐
        │  Standby Web Server SPA   │    │   Crash Forensics RTC     │
        │ • 80% экономия DRAM       │    │  • Запись логов сбоев     │
        │ • PROGMEM GZIP ресурсы    │    │  • Восстановление логов   │
        └───────────────────────────┘    └───────────────────────────┘
```

*   🛡️ **Zero-Allocation Hotpaths (Предсказуемость RAM):** В основном цикле исполнения (`loop`) полностью исключено динамическое выделение памяти (`malloc`/`new`). Все структуры инициализируются при запуске, а маршрутизация событий работает на основе O(1) хэшей FNV-1a. Это полностью предотвращает фрагментацию кучи и зависания.
*   📡 **Децентрализованный Mesh-Рой (Offline-First):** Самоорганизующаяся сеть на базе протокола `painlessMesh` объединяет узлы в единый вычислительный кластер. Сеть функционирует автономно без Wi-Fi роутеров, серверов и координаторов. Потеря части узлов компенсируется автоматической перемаршрутизацией трафика.
*   🔋 **Энергосберегающий HAL (Power-Stabilization Switch):** Поддержка физического отключения питания датчиков через транзисторные ключи (`powerPin`) во время спящего интервала. Прошивка бережно контролирует стабилизацию питания (`powerStabilizeMs`) перед съемом показаний, минимизируя окисление электродов и экономя батарею.
*   🌐 **On-Demand Веб-Интерфейс (Standby Mode):** Полноценное PWA SPA-приложение весом 250 КБ сжато до 46 КБ методом Pre-GZIP в памяти программ Flash (`PROGMEM`), обходя медленную LittleFS. Сервер автоматически засыпает при отсутствии клиентов, экономя драгоценную оперативную память микроконтроллера.
*   ⬛ **Бортовой Самописец (Post-Mortem BlackBox RTC):** При критическом сбое (Kernel Panic, срабатывание Watchdog Timer или просадка питания Brownout) контекст падения мгновенно пишется в защищенную RTC SRAM память. После перезапуска инженер может считать дамп командой `blackbox rtc` для детального анализа причин.

## 🌐 Веб-интерфейс и режим Standby

Одной из сложнейших задач при разработке прошивок для ESP32 является совмещение тяжелого сетевого стека, шифрования и веб-сервера в ограниченном объеме оперативной памяти (DRAM). AgriSwarm решает эту проблему с помощью оригинального механизма гибридной активности.

### 🗜️ Пре-компрессия веб-ресурсов (PROGMEM GZIP)
Вместо хранения статических файлов на файловой системе LittleFS, которая нагружает процессор дисковым вводом-выводом и расходует циклы перезаписи Flash, все файлы интерфейса (HTML, JS, CSS, PWA Manifest, Service Worker) предварительно сжимаются с помощью специального скрипта (доступного в составе SDK разработчика) и вшиваются в код как бинарные массивы:
*   Оригинальный размер: **247,6 КБ**
*   Размер GZIP в Flash (`PROGMEM`): **46,2 КБ** (сжатие на ~80%)
*   Браузер получает сжатые данные напрямую, благодаря HTTP-заголовку `Content-Encoding: gzip`. Поддержка `ETag` и жесткого кэширования исключает повторные запросы со стороны браузера.

### 💤 Режим ожидания (Standby Mode)
Веб-интерфейс устройства работает по принципу **On-Demand (по требованию)**:
```
           [Узел Запущен] ──► Режим Standby (Потребление RAM ~0)
                                    │
                                    │ (Запрос браузера на http://IP/)
                                    ▼
       Сервер отдает 1.5 КБ standby_html ──► [Кнопка АКТИВИРОВАТЬ]
                                                      │
                                                      │ (Клик / POST /api/server/start)
                                                      ▼
   Переход в Active Mode ◄────────────────────────────┘
   • Подключение SSE (/events) для логов в реальном времени
   • Регистрация коллбэков датчиков и пинов
   • Запуск API-роутов
        │
        ├─► [Активная работа пользователя в UI]
        │
        └─► (5 минут бездействия / таймаут)
                 │
                 ▼
     Автоматический сброс коллбэков, закрытие SSE, переход в Standby
```
1.  **Standby Mode (По умолчанию):** Запущен только базовый веб-сервер. Тяжелые JSON API, обработчики Server-Sent Events (SSE) и регулярные рассылки метрик отключены. При попытке зайти на сайт отдается минималистичная HTML-страничка весом в 1.5 КБ со спящим интерфейсом и кнопкой пробуждения.
2.  **Active Mode (Активный режим):** При клике на "Активировать" отправляется запрос к `/api/server/start`. Сервер разворачивает полноценное API, подключает SSE-поток `/events` для трансляции логов ядра и мгновенно связывается с коллбэками `PinManager`.
3.  **Авто-засыпание:** Если в течение 5 минут от веб-интерфейса не поступает запросов активности, веб-сервер принудительно засыпает, освобождая дескрипторы сокетов и очищая системные очереди.

### 📲 Web-OTA (Беспроводная прошивка через браузер)
В активном режиме доступен защищенный эндпоинт `/api/update` (HTTP POST). Вы можете скомпилировать бинарный файл (`firmware.bin`) в VS Code/PlatformIO и обновить прошивку любого узла через браузер в 2 клика без физического подключения программатора.

## 💻 Интерактивный CLI и мастера настройки

Для локальной отладки и быстрой конфигурации в полевых условиях разработан мощный командный интерфейс (CLI), доступный через Serial Monitor (115200 baud) или через вкладку CLI в веб-интерфейсе. 

CLI поддерживает **98 встроенных команд**, историю ввода (клавиши ↑/↓) и автодополнение по кнопке **Tab**.

### 🧙‍♂️ Встроенные мастера (Wizards)

Вам больше не нужно помнить синтаксис команд. В AgriSwarm добавлены три пошаговых интерактивных мастера:

1.  `wizard first_run` — Мастер первичной настройки. Спросит имя устройства, задаст параметры Mesh-сети (SSID/пароль) и выберет сетевую роль узла.
2.  `pin_wizard` — Конфигуратор железа. Безопасно проведет вас через привязку датчиков, реле или сервоприводов к физическим GPIO ESP32 с проверкой на недопустимые пины.
3.  `rule_wizard` — Визуальный конструктор автоматизации. Позволяет связать датчик с исполнительным устройством логикой IF-THEN, настроить кулдауны срабатывания и пороговые значения (Deadband).

## 🔌 Управление периферией и HAL

Физический слой управления в AgriSwarm абстрагирован от конкретного железа через `PinManager`. Система поддерживает широкий спектр аналогового и цифрового оборудования.

```
                  ┌──────────────────────┐
                  │      PinManager      │
                  └──────────┬───────────┘
                             │
       ┌─────────────────────┼─────────────────────┐
       ▼                     ▼                     ▼
┌──────────────┐      ┌──────────────┐      ┌──────────────┐
│ PIN_TYPE_DHT │      │  PIN_ANALOG  │      │  PIN_SERVO   │
│ • DHT11/22   │      │ • EMA фильтр │      │ • Угол 0-180°│
│ • Авто-сплит │      │ • PowerPin   │      │ • AutoDetach │
└──────────────┘      └──────────────┘      └──────────────┘
```

### 🌡️ Интеллектуальный опрос DHT11 / DHT22
При регистрации одного датчика DHT на одном GPIO, прошивка автоматически регистрирует два виртуальных системных канала: `имя_датчика/temp` и `имя_датчика/hum`. Опрос выполняется асинхронно, предотвращая блокировку FreeRTOS-потоков медленным протоколом 1-Wire.

### ⚡ Аппаратный ключ питания сенсоров (Борьба с окислением)
Популярные аналоговые датчики влажности почвы при постоянном нахождении под напряжением подвергаются процессу электролиза, из-за чего полностью разрушаются за несколько недель. 
*   **Решение в AgriSwarm:** Подключение плюса питания датчика к свободному GPIO (или через MOSFET-ключ).
*   **Логика работы:**
    1.  Запуск таймера опроса (например, каждые 30 минут).
    2.  Подача `HIGH` на `powerPin`.
    3.  Ожидание `powerStabilizeMs` (пока датчик выйдет на рабочий режим после подачи питания).
    4.  АЦП-измерение с наложением экспоненциального сглаживания (EMA-фильтр).
    5.  Подача `LOW` на `powerPin` (полное обесточивание датчика).

### ⚙️ Сервоприводы промышленного класса (`PIN_TYPE_SERVO`)
Для автоматизации вентиляционных заслонок, оконных фрамуг или дозаторов корма внедрена поддержка сервоприводов:
*   Калибровка минимального/максимального углов (0°–180°) и гибкая регулировка скорости движения.
*   **Функция Auto-Detach:** После завершения маневра сервопривод программно отключается от генерации ШИМ (`servoAutoDetach`). Это полностью устраняет раздражающий "зуд" и микровибрации сервопривода под нагрузкой, снижает энергопотребление и продлевает ресурс шестеренок редуктора.

## 📊 Технические характеристики

### 📐 Системные требования и лимиты
*   **Аппаратная база:** ESP32-WROOM-32D / ESP32-WROVER (240 MHz, Dual-Core).
*   **Память:** Не менее 4 МБ Flash, 520 КБ SRAM.
*   **Дисковая разметка:** LittleFS (раздел под файлы конфигурации JSON не менее 256 КБ).
*   **Максимальное число узлов в Mesh:** 20 (рекомендованная емкость для исключения коллизий — 10 узлов).
*   **Лимиты правил автоматики:** до 32 правил на один узел (ограничено статической хэш-таблицей).

### 📈 Объем кодовой базы
*   **Общий объем кода:** 32,767+ строк (27,278 .cpp + 5,489 .h).
*   **Архитектурных компонентов:** 89 файлов (модули логики, сети, драйверы HAL, утилиты).
*   **Документация проекта:** 31 глубокий технический документ.

## 🛡️ Отказоустойчивость и безопасность

Поскольку физические устройства работают в агрессивных средах (влажность, перепады температур, нестабильное электроснабжение), в AgriSwarm заложены продвинутые оборонительные механизмы:

1.  **Двойной аппаратный WDT:** Независимый сброс зависших задач FreeRTOS. Если задача сети заблокирует ядро более чем на 8 секунд, Stability Manager принудительно перезагрузит устройство.
2.  **Защита по питанию (Brownout Detector):** Отслеживание просадок входного напряжения. При падении напряжения ниже 2.7В система сохраняет критические данные конфигурации и останавливает периферию, предотвращая порчу памяти LittleFS.
3.  **Изоляция сбоев (Safe Wrappers):** Опрос внешних датчиков обернут в безопасные обработчики исключений. Потеря контакта с DHT22 или КЗ на аналоговой линии не вешают операционную систему, узел продолжает маршрутизировать Mesh-трафик.
