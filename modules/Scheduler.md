# ⏱️ Модуль Scheduler (src/Scheduler.cpp)

## 1. Обзор модуля и его роль в системе
Класс `TaskScheduler` — это легковесный диспетчер периодических задач. Он отвечает за выполнение определенных действий (команд) через заданные интервалы времени. Это аналог `cron` для встраиваемых систем, но работающий в рамках кооперативной многозадачности (внутри `loop()`).

**Основная роль:**
*   Регистрация периодических фоновых задач (например, "опрашивать датчик каждые 10 секунд" или "отправлять пинг раз в минуту").
*   Поддержка паузы, возобновления и принудительного (ручного) запуска задач.
*   Управление списком задач через `ConfigManager` (сохранение в `tasks.json`).

## 2. Зависимости
*   `ConfigManager` — для сохранения и загрузки конфигурации (`tasks.json`).
*   `HashUtils.h` — для хеширования идентификаторов задач (оптимизация поиска).
*   `FreeRTOS (semphr.h)` — потокобезопасность через мьютексы.

## 3. Анализ структур данных и паттернов
*   **Структура `ScheduledTask`**: Содержит строковый ID, текстовую команду (`command`), интервал в миллисекундах (`intervalMs`), время последнего запуска и флаг активности.
*   **Оптимизация поиска (ФАЗА 3):** Внутри `TaskScheduler` задачи хранятся в векторе структур `HashTask`, который отсортирован по `hash`. Это позволяет использовать бинарный поиск (`std::lower_bound`) для O(log N) доступа к задаче по её строковому ID, что критично при большом количестве мелких тасок.
*   **Потокобезопасность:** Добавлен рекурсивный мьютекс `SemaphoreHandle_t _mutex`. Все публичные методы (add, remove, get, update) обернуты в RAII-локер `TaskLock`. Это позволяет безопасно модифицировать расписание из разных потоков (например, при получении команды по сети).

## 4. Анализ логики выполнения (`update`)
Метод `update()` вызывается из `SystemOrchestrator` в главном цикле.
1.  Перебирает вектор `_tasks`.
2.  Если задача `active` и прошло достаточно времени (`millis() - task.lastExecutionTime >= task.intervalMs`), она ставится на выполнение.
3.  Выполнение делегируется наружу через вызов `_commandCallback(task.command)`. Сам планировщик не знает, что означает команда (это может быть CLI-инструкция или вызов функции в другом менеджере). Обычно коллбэк указывает на `CommandExecutor::executeCommand()`.
4.  После успешного выполнения обновляется `lastExecutionTime`.

## 5. Выводы
`TaskScheduler` — это простая, но высокооптимизированная реализация диспетчера задач. Использование бинарного поиска по хешам и рекурсивных мьютексов делает его безопасным и быстрым. Делегирование исполнения через `std::function` позволяет полностью отвязать планировщик от бизнес-логики проекта (он не знает, как дергать пины или слать сеть, он просто передает строку-команду).
