# 📡 Раздел 4: Mesh Network and Data Routing

Описание работы децентрализованной сети AgriSwarm v4.0.4, алгоритмов маршрутизации и самовосстановления.

## 1. Топология и Роли (Topology & Roles)

Сеть строится по принципу однорангового (P2P) взаимодействия на базе Wi-Fi Mesh (painlessMesh).

**Роли узлов:**
*   **SENSOR:** Конечный узел, только передача данных.
*   **ACTUATOR:** Исполнитель, прием команд.
*   **HYBRID:** Сенсор + Актуатор (наиболее частая роль).
*   **GATEWAY:** Шлюз, мост в интернет/локальную сеть.

### ⚠️ ВНИМАНИЕ: Ограничения по емкости узла
В версии v4.0.4 поиск в кэшах данных выполняется линейно.
*   **Рекомендация:** При планировании сети с большим количеством датчиков (100+) распределяйте нагрузку по нескольким узлам-агрегаторам для предотвращения задержек в обработке правил.

## 2. Маршрутизация (Routing)

**Передача телеметрии (Sensor Data):**
Используется управляемая flood-рассылка (Broadcast). Пакеты содержат `messageId` и `seq` для предотвращения "широковещательных штормов". Узлы-ретрансляторы отбрасывают дубликаты на основе кэша хэшей.

**Команды управления (Reliable Unicast):**
Отправляются конкретному узлу с подтверждением доставки (ACK).
*   Используется `MeshDeliveryTracker` для контроля таймаутов.
*   **Retry Policy:** 3 попытки отправки. Если ACK не получен за 5 секунд, пакет переотправляется (параметр `_maxRetries`).
*   **Резервирование команд (CommandBackupSystem):** Если узел-получатель временно перегружен или недоступен (обрыв связи), модуль `CommandBackupSystem` сохраняет критические команды в локальный кэш и гарантированно доставляет их, как только связь восстановится.

## 3. Адаптивные профили связи (MeshOptimizer)
Для поддержания максимальной стабильности в меняющихся условиях среды (дождь, открытое поле, низкий заряд батареи) система использует динамическую адаптацию через модуль `MeshOptimizer`.
*   Администратор может на лету применять профили `mesh_preset` (ECO, FAST, STABLE) без перезагрузки всей логики.
*   Команда `mesh_advanced weather rain` принудительно повышает мощность передатчика для компенсации затухания радиоволн в каплях воды во время дождя.

## 4. Синхронизация времени

Узлы синхронизируют внутренние счетчики времени (`_mesh.getNodeTime()`) через beacon-фреймы. 
*   Все `timestamp` в сети относительны и базируются на синхронизированном аптайме роя.
*   Gateway-узел может выступать эталоном времени, если имеет доступ к NTP.

## 5. Самовосстановление (Self-Healing)

Система непрерывно мониторит качество соединений через `SmartMeshManager`.
1.  **Детекция:** Если узел-сосед не отвечает 30 сек, он помечается как `DISCONNECTED`.
2.  **Поиск пути:** Узел ищет альтернативного "родителя" среди доступных соседей.
3.  **Критерии выбора:** 40% вес задержки (Ping) + 30% уровень сигнала (RSSI) + 30% загруженность соседа (Load).
4.  **Fallback:** Если связь полностью потеряна, через 60 сек узел поднимает аварийную точку доступа `Host Mode AP` для ручной диагностики.
