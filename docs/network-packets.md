# 📡 Спецификация сетевых JSON-пакетов (Network Packet Specification)

В децентрализованной сети AgriSwarm v4.0.4 обмен данными между узлами в Mesh-сети (на базе `painlessMesh` / ESP-NOW) осуществляется в текстовом формате с использованием сжатых JSON-объектов.

Этот документ содержит спецификацию JSON-схем для всех типов пакетов, циркулирующих в сети. Это необходимо для написания внешних шлюзов (например, скриптов на Python / Node-RED для Raspberry Pi), шлюзов MQTT или интеграции с Home Assistant.

---

## 🧭 Глобальные служебные поля

Каждое сетевое сообщение содержит стандартные поля:
*   `type` (String) — Тип пакета (используется для маршрутизации в `MessageRouter`).
*   `messageId` (unsigned long) — Уникальный инкрементный идентификатор сообщения на узле-отправителе (для ACK подтверждений).
*   `seq` (unsigned long) — Номер последовательности для защиты от дублирования сообщений при лавинной маршрутизации (Broadcast Storms).

---

## 📋 Спецификация пакетов по типам

### 1. 🌡️ Телеметрия датчиков (`sensor_data`)
Публикуется узлом в режиме Broadcast при изменении значения датчика (с учетом порога нечувствительности deadband) или периодически по таймеру (Heartbeat).

**Пример пакета:**
```json
{
  "type": "sensor_data",
  "sensorId": "soil_analog",
  "value": 28.5,
  "unit": "%",
  "topic": "agriswarm/sensor/3284719/soil_analog",
  "messageId": 1042,
  "seq": 25,
  "isValid": true,
  "timestamp": 1248590
}
```

**Описание полей:**
*   `sensorId` — уникальное имя датчика на узле.
*   `value` — физическое значение датчика (float).
*   `unit` — единица измерения (`C`, `%`, `V`, `Pa` и т.д.).
*   `topic` — сгенерированный MQTT-совместимый топик датчика.
*   `isValid` — статус достоверности показания (false при сбоях чтения датчика).
*   `timestamp` — локальный аптайм отправителя в миллисекундах.

---

### 2. ⚡ Команда управления актуатором (`actuator_command`)
Отправляется на конкретный целевой узел (Unicast) для переключения реле или поворота сервопривода. Требует подтверждения (ACK).

**Пример пакета:**
```json
{
  "type": "actuator_command",
  "actuatorId": "relay_pump",
  "command": "1.0",
  "commandId": 59201,
  "requireAck": true,
  "timestamp": 1250100,
  "crc": "d5f6a2b8"
}
```

**Описание полей:**
*   `actuatorId` — имя исполнительного устройства на целевой плате.
*   `command` — строка команды (`"1.0"` — включить реле, `"90"` — повернуть сервопривод).
*   `commandId` — уникальный ID команды (генерируется на основе eFuse MAC-адреса и аптайма).
*   `requireAck` — требовать подтверждения получения (будет отправлен `actuator_ack`).
*   `crc` — CRC32 чек-сумма для валидации данных (`MessageCRC`).

---

### 3. ℹ️ Объявление возможностей исполнителя (`actuator_info`)
Узлы периодически рассылают информацию о доступных актуаторах и их характеристиках.

**Пример пакета:**
```json
{
  "type": "actuator_info",
  "nodeId": 3284719,
  "actuatorId": "relay_pump",
  "actuatorType": "RELAY",
  "isAvailable": true,
  "value": 0.0,
  "requiredLevel": 2,
  "description": "Клапан автополива",
  "lastUpdate": 1249800,
  "topic": "agriswarm/actuator/3284719/relay_pump"
}
```

**Описание полей:**
*   `actuatorType` — тип устройства (`RELAY`, `SERVO`, `DIGITAL_OUT`).
*   `requiredLevel` — уровень прав для доступа (например, 2 = Advanced, 4 = Admin).
*   `isAvailable` — готовность устройства к работе.

---

### 4. 💓 Маяки присутствия (`heartbeat`)
Каждый узел периодически отправляет Broadcast для информирования о своем присутствии, поддержания топологии сети и синхронизации времени.

**Пример пакета:**
```json
{
  "type": "heartbeat",
  "nodeId": 3284719,
  "timestamp": 1251000
}
```

---

### 5. 👥 Запрос системных параметров узла (`node_info`)
Используется для удаленного мониторинга и получения детальной технической информации о здоровье конкретной платы.

**Запрос (Unicast):**
```json
{
  "type": "node_info_req",
  "timestamp": 1252000
}
```

**Ответ (Response Unicast):**
```json
{
  "type": "node_info_resp",
  "nodeId": 3284719,
  "name": "greenhouse-1",
  "role": "hybrid",
  "uptime": 1252,
  "freeHeap": 184520,
  "minFreeHeap": 124110,
  "health": 100.0,
  "loopTimeMs": 3,
  "resetReason": "POWERON_RESET"
}
```

---

### 6. 🧪 Проверка задержки связи (`ping` / `pong`)
Используется для расчета качества соединения (`MeshPerformanceMonitor`) и времени отклика в сети (RTT).

**Сообщение Ping:**
```json
{
  "type": "ping",
  "senderId": 3284719,
  "timestamp": 1253000
}
```

**Сообщение Pong:**
```json
{
  "type": "pong",
  "senderId": 3285112,
  "timestamp": 1253012
}
```
*Задержка вычисляется как разность между локальным временем отправки и временем получения ответа.*

---

## 🛡️ Безопасность и Целостность пакетов

1.  **Контроль CRC32:** Для всех управляющих команд (`actuator_command`) в поле `crc` записывается 32-битная хэш-сумма. Если CRC в пришедшем пакете не совпадает с вычисленной принимающим узлом — команда сбрасывается с ошибкой `INVALID_CRC` в `MessageCRC`.
2.  **Белый список (`TrustedNodeManager`):** Перед выполнением любой команды узел проверяет отправителя по таблице MAC-адресов. Запросы от неавторизованных `nodeId` игнорируются, защищая периметр от несанкционированного управления реле и кражи данных.
