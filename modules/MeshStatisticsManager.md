# 📊 Класс: MeshStatisticsManager

## 1. Назначение и роль в системе
Класс `MeshStatisticsManager` предназначен для сбора, накопления и сохранения долгосрочной статистики о поведении узлов в mesh-сети. Он ведет учет количества подключений, разрывов, стабильности и качества линков (RSSI), а также обеспечивает сохранение собранных данных во встроенную Flash-память (LittleFS) с защитой от быстрого износа ячеек памяти.

## 2. Метрики узлов (`MeshNodeStats`)
Каждая запись о состоянии узла делится на два блока:

### 2.1. Долгосрочные данные (За все время)
*   `firstSeenMs` / `lastSeenMs` — временные метки первого и последнего контакта.
*   `connectionCount` / `disconnectCount` — счетчики подключений и разрывов линка.
*   `connectedTimeTotalMs` — общее время нахождения узла в сети.
*   `lastKnownRSSI` — последний уровень сигнала (dBm).
*   `rssiSamples` — кольцевой буфер из последних 20 измерений RSSI для сглаживания шумов.
*   `stabilityScore` — интегральная оценка стабильности связи (0..100%).
*   `lastGoodDurationMs` — длительность самого долгого периода беспрерывной связи.

### 2.2. Метрики текущей сессии
*   `sessionStartMs` — время запуска текущего сеанса.
*   `sessionConnectionCount` / `sessionDisconnectCount` — статистика соединений за сеанс.
*   `sessionStabilityScore` — локальная оценка стабильности в текущем сеансе.

## 3. Защита Flash от износа (Wear Leveling)
Поскольку запись во Flash-память ESP32 при частых событиях подключения/отключения может быстро исчерпать ресурс циклов перезаписи (обычно 10 000–100 000 циклов), в `MeshStatisticsManager` реализован буферизованный алгоритм сохранения:
*   `SAVE_INTERVAL_MS` = `300000` (5 минут) — интервал фонового автосохранения.
*   `MIN_SAVE_INTERVAL_MS` = `30000` (30 сек) — минимальная задержка между принудительными сохранениями.
*   `CHANGES_THRESHOLD` = `10` — запись инициируется только после накопления 10+ изменений.
*   Экстренное сохранение `emergencySave()` срабатывает при критических системных событиях.

## 4. Основные функции и интерфейсы
*   `onNewConnection` / `onDroppedConnection` — триггеры фиксации событий.
*   `onHeartbeat` / `onMessageFrom` — фиксация уровней RSSI при радиообмене.
*   `getNodeReport` — генерация текстового отчета о состоянии линка узла.
*   `pruneStaleNodes` — очистка неактивных узлов (старше 48 часов) для предотвращения раздувания файлов.

## 5. Zero-Alloc Потоковая Запись (Streaming)
В версии v4.0.4.2 модуль переведен на парадигму Zero-Alloc Hotpath для процесса сохранения файлов:
*   Функция `saveStatistics()` больше не использует `DynamicJsonDocument` для сборки большого JSON-дерева в оперативной памяти.
*   Вместо этого применяется **потоковая сериализация** (direct streaming): массивы и объекты формируются на лету строковыми операциями и отправляются напрямую в класс `File` (`file.print()`).
*   Это полностью устранило пиковые всплески фрагментации кучи (Heap Fragmentation) при сохранении статистики десятков узлов, что было частой причиной зависаний в предыдущих версиях.
