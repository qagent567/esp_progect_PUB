# ⌨️ Модуль CommandExecutor (src/CommandExecutor.cpp)

## 1. Обзор модуля и его роль в системе
`CommandExecutor` — это подсистема взаимодействия устройства с оператором (Командная строка / CLI). В условиях децентрализованной сети ESP32 часто настраивается не через Web-интерфейс, а через прямое подключение по Serial (UART).

**Основная роль:**
*   Предоставление интерактивной консоли с приглашением вида `agriswarm@Node1$ `.
*   Поддержка истории команд (навигация стрелками вверх/вниз) и автодополнения по клавише `Tab`.
*   Обработка пользовательского ввода с клавиатуры (включая невидимые символы и backspace).
*   Маршрутизация команд в специализированные обработчики (`SystemCommandHandler`, `NetworkCommandHandler` и т.д.).

## 2. Архитектура и Рефакторинг
В коде присутствует явное архитектурное предупреждение: `// ⚠️ АРХИТЕКТУРНОЕ ПРЕДУПРЕЖДЕНИЕ: Этот файл находится в процессе рефакторинга`.
Ранее этот класс представлял собой огромный монолитный `if-else` блок на тысячи строк. Сейчас он переходит на архитектуру **Chain of Responsibility (Цепочка обязанностей)**:
*   В конструкторе создаются "хендлеры": `_systemHandler`, `_networkHandler`, `_hardwareHandler` и др.
*   Метод `_executeCommand` просто передает строку по цепочке: `if (_systemHandler->tryHandle(cmd, args)) return;`. Если ни один обработчик не забрал команду, выводится ошибка. Это радикально упростило поддержку и расширение CLI.

## 3. Анализ логики (Специфика встраиваемых систем)

### 3.1. Асинхронный CLI Task
Чтение символов из порта (`Serial.read()`) — это медленная операция (IO Bound). Если поместить её в главный цикл `loop()`, ожидание ввода пользователем заблокировало бы сетевой стек и датчики.
Поэтому `CommandExecutor` выделен в **отдельную задачу FreeRTOS**:
* **Параметры задачи FreeRTOS**:
  * **Название задачи:** `cliTask` (выделенный поток FreeRTOS)
  * **Размер стека:** 8 КБ (8192 байт)
  * **Приоритет задачи:** 2 (фоновое выполнение)
  * **Ядро процессора:** Запуск на выделенном ядре ESP32
Эта задача бесконечно крутится в фоне (`_cliTask`), собирает символы по одному, обрабатывает Backspace и ANSI Escape-коды (стрелки), и только когда нажат `Enter` — передает собранный буфер на исполнение. Использование `vTaskDelay(20)` отдает кванты времени другим задачам ядра.

### 3.2. Интеграция с UI (NotificationSystem & Logger)
В терминале ESP32 вывод команд и фоновых логов происходит в один поток (Serial). Если оператор набирает команду `mesh_reboot`, а в этот момент прилетает лог `[INFO] Датчик влажности 40%`, текст перемешается: `mesh[INFO] Датчик влажности 40%_reboot`.
Для решения этой проблемы модуль интегрирован с:
*   `LogPauseManager`: Если пользователь начал печатать (в буфере есть символы), вывод фоновых логов временно ставится "на паузу" (мьютится), чтобы не мешать набору.
*   `NotificationSystem`: Если на экране висит красное окно "КРИТИЧЕСКАЯ ОШИБКА", любое нажатие клавиши оператором сначала закроет это окно, и только затем появится промпт ввода команд.

### 3.3. Защита от переполнения
Буфер ввода жестко ограничен (`COMMAND_BUFFER_SIZE`). Если пользователь зажмет кнопку на клавиатуре, буфер не переполнится (что вызвало бы buffer overflow и хакинг стека), а выдаст ошибку `⚠️ Превышен лимит символов в команде!`, очистит буфер и покажет пустой промпт.

## 4. Выводы
Модуль `CommandExecutor` предоставляет "взрослый" опыт использования консоли (с историей, табами и поддержкой ANSI-стрелок), сравнимый с bash, но работающий на микроконтроллере с 300 КБ ОЗУ. Вынесение парсинга клавиш в отдельную задачу FreeRTOS изолирует интерфейс пользователя от критической бизнес-логики (RuleEngine).
