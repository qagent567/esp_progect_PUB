# 1. Обзор проекта

## Назначение и решаемая проблема
**AgriSwarm** — это децентрализованная программная платформа (прошивка) для автоматизации сельского хозяйства на базе микроконтроллеров ESP32. Проект решает проблему отсутствия надежной связи в полевых условиях, предлагая автономную Mesh-сеть, которая работает без центрального сервера, интернета или облака. Прошивка ориентирована на создание отказоустойчивых систем управления (полив, освещение, мониторинг) в условиях сложного рельефа и отсутствия инфраструктуры (Offline-first).

## Целевая аудитория
- **Технический сегмент (IoT-инженеры)**: Специалисты, создающие кастомные системы автоматизации. Для них проект предлагает гибкий API, расширяемую систему драйверов и мощный инструментарий диагностики.
- **Бизнес-сегмент (Агрохолдинги и фермеры)**: Пользователи, которым важна простота развертывания и надежность. Система позволяет масштабировать сеть простым добавлением новых узлов без перенастройки всей инфраструктуры.

## Ключевые отличия от аналогов
- **Децентрализация (P2P)**: Отсутствие «главного» узла; при выходе из строя любого элемента сеть перестраивается автоматически.
- **Аппаратная форензика**: Встроенный «Чёрный ящик» (BlackBox), хранящий состояние системы в RTC-памяти. Это позволяет узнать причину краша даже после перезагрузки.
- **Нулевая зависимость от облаков**: Все правила, задачи и алгоритмы исполняются локально на узлах.
- **Производительность**: Использование FNV-1a хеширования для мгновенного (O(1)) поиска топиков и правил в памяти.

# 2. Архитектура и структура кода

## Детальный разбор директории `include/` (Публичные интерфейсы)
- **`SystemOrchestrator.h`**: Главный мозг системы. Управляет инициализацией и связями между всеми менеджерами.
- **`AgriNetworkManager.h`**: Управление гибридной сетью. Обертка над `painlessMesh` с логикой автовосстановления.
- **`PinManager.h`**: Абстракция аппаратного уровня. Работает через интерфейс `IPinDriver`.
- **`RuleEngine.h`**: Реактивная логика. Позволяет связывать события от датчиков с действиями актуаторов.
- **`EventManager.h`**: Центральная Pub/Sub шина для обмена данными внутри прошивки.
- **`Structures.h`**: Единый источник правды для структур данных (состояния пинов, сетевые пакеты, роли узлов).
- **`Constants.h`**: Глобальные настройки (версия v4.0.4, порты, таймауты, логирование).

## Детальный разбор директории `src/` (Реализация)
- **Модульность**: Проект построен по паттерну «Менеджер». Каждый менеджер (Network, Pin, Rule, Config) отвечает за свой домен и минимально зависит от других.
- **Драйверная модель (`src/drivers/`)**:
    - `AnalogDriver.cpp`: Работа с АЦП, поддержка «пина питания» для экономии энергии.
    - `DhtDriver.cpp`: Чтение температуры и влажности (DHT11/22) с защитой от ошибок чтения.
    - `ServoDriver.cpp`: Плавное управление углами, автоотключение (detach) для экономии заряда.
- **Обработчики команд (`src/handlers/`)**:
    - `AutomationCommandHandler.cpp`: Огромный модуль (80КБ+), реализующий CLI для управления правилами и алгоритмами.

## Карта взаимодействия компонентов
1. **Hardware** -> `PinManager` (опрос) -> `EventManager` (публикация).
2. `EventManager` -> `RuleEngine` (сравнение с условиями) -> **Action**.
3. **Action** -> `PinManager` (локальное исполнение) ИЛИ `NetworkManager` (отправка на другой узел).

## Внешние зависимости
- **ArduinoJson**: v6.19.0.
- **painlessMesh**: Для построения самоорганизующейся сети.
- **CRC / MessageCRC**: Собственная реализация проверки целостности данных.

# 3. Функционал и настройки

## Список реализованных функций
- **Mesh-связь**: Автоматическое объединение до 20+ узлов в единую сеть.
- **Pub/Sub Топики**: Поддержка топиков в формате `agriswarm/<nodeId>/<sensorId>`.
- **Реактивные правила**: Поддержка условий `>`, `<`, `==`, `!=`, `BETWEEN`, `CHANGED`.
- **Сложные алгоритмы**: `AlgorithmScheduler` поддерживает цепочки действий с задержками (`DELAY`, `REPEAT`).
- **Интерактивный CLI**: Более 100 команд для полной настройки через Serial/Terminal.
- **Система калибровки**: Команды `servo_calibrate` и `pin_calibrate` для настройки оборудования "на лету".

## Параметры конфигурации
- **Файлы**: `/config.json` (основной), `/rules.json` (автоматизация), `/wifi_config.json` (внешний Wi-Fi).
- **Сетевые пресеты**:
    - `STABLE`: Медленный Heartbeat, высокая устойчивость к помехам.
    - `FAST`: Быстрый отклик для тестирования.
    - `ECO`: Максимальное энергосбережение.

## Статус функций
- **Работает стабильно**: Базовая Mesh-связь, драйверы Relay/Digital/Analog, CLI, RuleEngine.
- **В бете**: `AlgorithmScheduler` (функции `algo_create` в процессе разработки), `SmartMesh` (динамический выбор каналов).
- **TODO**: Шифрование пакетов, OTA-обновление всей сети через Mesh.

# 4. Честная техническая оценка

## Сильные стороны
- **Исключительная диагностика**: Благодаря BlackBox и CLI можно восстановить последовательность событий до краша.
- **Эффективность памяти**: Минимальное использование `std::string` в горячих путях, переход на хэш-ключи.
- **Гибкость**: Настройка практически любого аспекта системы (таймауты, интервалы, пины) без перепрошивки.

## Ограничения и недостатки
- **JSON Overhead**: Использование JSON для Mesh-сообщений создает нагрузку на CPU при большом трафике.
- **Технический долг**: Огромные файлы-обработчики (`AutomationCommandHandler.cpp`) затрудняют навигацию и требуют рефакторинга.
- **Масштабируемость**: Эффективная работа Mesh ограничена ~30 узлами из-за ограничений протокола и памяти ESP32.

## Известные риски
- **Brownout**: Пиковое потребление ESP32 при передаче Mesh-пакетов может вызвать перезагрузку при слабом питании.
- **Фрагментация Heap**: Динамическое создание правил в RuleEngine может привести к фрагментации памяти при длительной работе.

# 5. Сценарии практического применения

## Реальные use-cases
- **Автоматизированный полив**: Датчики влажности почвы на разных узлах управляют центральным насосом.
- **Система оповещения**: Датчик протечки/дыма на одном конце фермы мгновенно зажигает тревожный LED на всех остальных узлах.
- **Умное освещение**: Управление группами реле по расписанию (`TaskScheduler`) или датчику освещенности.

## Типичные ошибки пользователей
- **Конфликт GPIO**: Попытка использовать один и тот же пин для разных типов драйверов.
- **Забивание сети**: Настройка слишком частого опроса датчиков (менее 100мс) на всех узлах сразу.
- **Игнорирование `pin_save`**: Настройка пина через CLI без вызова `pin_save` приведет к сбросу настроек после перезагрузки.

# 6. Подготовка к публикации

## Требования к сборке
- **PlatformIO**: Рекомендуемый инструмент (конфиг `platformio.ini` в корне).
- **FileSystem**: Обязательная прошивка образа LittleFS.
- **Зависимости**: Библиотеки устанавливаются автоматически через PIO Library Manager.

## Рекомендации перед релизом
1. Очистить логи отладки (`CORE_DEBUG_LEVEL=0`).
2. Провести калибровку RTC-часов для корректных меток времени в BlackBox.
3. Проверить целостность файловой системы командой `fs_info`.

# 7. Опорные точки для маркетинга и коммуникации

- **Факты для позиционирования**:
    - "Система с памятью": BlackBox хранит историю ошибок даже после отключения питания.
    - "Mesh за 5 минут": Автоматическое объединение узлов без настройки маршрутизаторов.
    - "Промышленный стандарт автоматизации": Использование детерминированных правил вместо хрупких скриптов.
- **Честная трансляция ограничений**: Важно указывать, что система требует качественного питания (5V 1A min) для стабильной работы радиомодуля.
