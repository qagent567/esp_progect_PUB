# 📦 Модуль MessageBuffer (include/MessageBuffer.h)

## 1. Обзор модуля и его роль в системе
`MessageBuffer` — это специализированный In-Memory логгер. Он представляет собой легковесное хранилище последних текстовых сообщений системы (логов). 

**Основная роль:**
*   Буферизация текстовых событий системы в ОЗУ для последующего отображения (например, в CLI командой `history` или на Web-интерфейсе).
*   Предоставление истории для диагностики при разборе полетов (Post-Mortem).

## 2. Анализ структур данных и паттернов

### 2.1. Структура `LogMessage`
Хранит запись журнала:
*   `timestamp`: время (millis).
*   `message`: само сообщение (класс `String`).
*   `source`: источник ("NetworkManager", "RuleEngine" и т.д.).
*   `level`: уровень важности (INFO, WARN, ERROR).

### 2.2. Оптимизация: Кольцевой буфер (Ring Buffer)
Изначально хранилище могло быть реализовано через `std::vector`, но вызов `erase(begin())` при переполнении вектора сдвигает все элементы, что дает сложность O(N).
Модуль был переписан (ФАЗА 2) с использованием классического статического кольцевого буфера:
* **Кольцевой буфер в памяти**:
  * Буфер сообщений (`LogMessageMessages`) с фиксированным максимальным лимитом записей (`MAX_BUFFER_MESSAGES` = 50) и указателем на текущую «голову» буфера.
*   `MAX_BUFFER_MESSAGES` жестко задано как 50. 
*   При добавлении 51-го сообщения оно просто перезаписывает самое старое (по индексу `_head`), а индекс смещается `_head = (_head + 1) % 50`.
*   Сложность вставки: **O(1)**. Нет аллокаций массива, нет сдвигов. Вставка лога не тормозит систему.

## 3. Анализ логики

*   `addMessage()`: Добавляет лог в массив. Защищено от переполнений.
*   `getMessages()` / `getLastMessages(N)`: При вызове геттеров массив `_messages` копируется во временный вектор `_tempMessages` в правильном хронологическом порядке. Это ресурсоемкая операция (O(N)), но она вызывается только тогда, когда пользователь явно запрашивает логи (например, через терминал), что приемлемо, так как чтение не происходит в "горячем цикле" управления.
*   `filterBySource` / `filterByLevel`: Позволяет делать выборки логов (например, показать только ошибки).

## 4. Выводы и узкие места
Модуль крайне быстр на запись (O(1)). 
Единственное "узкое место" — использование класса `String` внутри массива `_messages`. При каждой записи происходит аллокация в куче для копирования содержимого строки. Для текстового логгера это терпимый компромисс, так как длина логов обычно невелика, однако при экстремальной нехватке памяти (Out of Memory) эта подсистема может пострадать первой. В идеале `String` следует заменить на фиксированные массивы `char[64]`.
